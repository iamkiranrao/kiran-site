# The Build Journey — From Certs Page to Skills System
**April 2, 2026**

---

## The Architectural Pivot

The learning page was a static grid of 12 certification cards. What shipped is a dual-mode interactive skills explorer with 33 skills, 16 evidence sources, ~50 link relationships, 6 color-coded domains, a force-directed graph powered by D3.js v7, and a JSON data model that makes adding a new skill a three-line change.

The pivot happened in one sentence: "Instead of certification cards. I'm thinking of representing the information in a different visual." From that point, the old page was dead and the new one had to be built from scratch in the same session.

---

## Six Prototypes, One Winner

The exploration phase produced six working prototypes in about 40 minutes:

1. **Constellation Map** — Skills as glowing star-map dots clustered by domain. Beautiful, but brittle at scale and terrible on mobile (tiny tap targets, no hover).
2. **Force-Directed Graph** — Skills AND sources as visible D3 nodes with physics simulation. Both directions work but visually noisy with 49 nodes.
3. **Grouped Tag Wall** — Skills as clickable pills grouped by domain, evidence drawer on click. Most scannable, most mobile-native, most extensible.
4. **Option A (Dual-mode)** — Tag wall + force graph with a toggle switch.
5. **Option B (Hybrid)** — Tag wall with SVG connection lines animating between related skills.
6. **Option C (Skill-centered graph)** — Only skills visible as nodes; sources are invisible "attractor" nodes creating physics-based clustering.

Kiran's pick: Option A's dual-mode structure with Option C's skill-centered graph engine. The combined prototype (`proto-combined.html`) became the foundation.

---

## The D3 Bug Gauntlet

The force graph broke multiple times across the session:

**Bug 1: Graph container zero height.** The `.graph-view` had `height: 100%` but its parent had no explicit height when the graph initialized. D3 created a zero-height SVG. Fix: explicit `min-height: 700px` plus `requestAnimationFrame` deferral.

**Bug 2: Link data type mismatch.** Links used numeric indices from `nodeMap.get()`, but `forceLink.id()` expected string IDs. D3 couldn't resolve link endpoints. Fix: switch links to use string IDs matching the `.id()` accessor.

**Bug 3: Temporal Dead Zone (TDZ).** This one was subtle. After a centering fix moved code around, `const width` and `const height` were declared on lines 1002-1003 but referenced on lines 956-957 in the node initialization. JavaScript's TDZ made these variables inaccessible before their declaration line — `const` is hoisted but not initialized. The entire `initializeGraphView()` function silently died. No error in the console because the graph init was wrapped in a try/catch-like flow. Fix: move dimension calculation above node creation.

The TDZ bug was the most educational. It's the kind of error that JavaScript developers encounter once and never forget, but it's easy to introduce when reorganizing code within a function. The lesson: when moving `const` declarations around, always check if anything above the new position references them.

## Production Integration: The Medium Is the Message

Merging `proto-combined.html` into production as `skills.html` required integrating with the site chrome: nav, mobile menu, persona badge, Fenix banner, footer, GA4, Clarity, OG tags. The prototype was a standalone dark-themed app. Production needed to coexist with elements that weren't in the prototype's universe.

Three visual regressions emerged immediately:

**1. Detail panel height.** The prototype used `position: fixed; top: 0; height: 100%` — the panel fills the viewport edge-to-edge. The first production attempt changed this to `position: absolute` within the content-area, which made the panel shorter. Kiran saw this in screenshots and said the prototype was better. Reverted to `position: fixed`.

**2. Footer visibility.** The prototype had no footer. Production inherited the site footer, breaking the full-viewport app feel. Fix: `footer, .toast { display: none !important; }`. The skills page IS the viewport — there's nothing below it.

**3. Domain header colors.** The prototype showed colored domain labels (PRODUCT in salmon, AI & ML in blue). Production had them in gray. The CSS defined `color: #555` for `.domain-label`, but the prototype's dark background made this semi-visible while production's slightly different background made it invisible. Fixed by setting `label.style.color = domain.color` directly in JavaScript during domain rendering.

## The Universal Rename

Changing "Learning & Certifications" → "Skills" touched 37 HTML files, plus `bento-cards.js` and `sitemap.xml`. The rename was surgical: only navigation links and URL references were changed, not CSS class names (`.slot-learning`, `[data-card="learning"]`) or content text mentioning "learning." The CSS classes are structural — renaming them would break the bento grid and persona system.

One tricky detail: HTML entities like `&#8600;` (↘ arrow) in navigation links couldn't be matched by sed or Python string replacement. The fix was using the Read tool to see exact file content, then the Edit tool with the exact character sequence.

## The Cleanup Phase

The continuation session was entirely about polish and cleanup:

- **Orange Fenix border** bleeding into the skills layout — caused by `.fenix-subpage-inner`'s `border-bottom: 1px solid var(--persona-accent)` in `styles.css`. First CSS override failed due to insufficient specificity (one class vs two). Fixed by matching the parent selector `.fenix-subpage-full .fenix-subpage-inner`.

- **Black bar gap** between header and skills grid — 65px of combined padding (24px header + 40px grid-view top) creating a visible dark band. Reduced to 36px total (16px + 20px).

- **Fenix widget removal** — fenix-widget.js and fenix-widget.css stripped from all 30 HTML files. The Fenix subpage banner (from persona-system.js) disabled by commenting out `initFenixSubpageModule()`. Both changes preserve the code for future re-enabling.

---

## What Shipped

`skills.html` is live at kiranrao.ai/skills — a full-viewport interactive skills explorer with dual-mode viewing (tag grid + force graph), a right-side detail panel showing evidence sources and connected skills, 6 color-coded domains, and a JSON data model that makes the page a living system rather than a static artifact. The old learning.html still exists for potential redirect handling.

The data architecture — skills → sources → links — is the real win. It transforms the page from a portfolio piece into an extensible knowledge graph. When Kiran earns a new cert or ships a new prototype, it's a few lines of JSON, not a layout redesign.

## Session 16-17: Add Skills Evidence Management + Mind the Gap

### The Evidence Management System

The second major build of the day was the entire Add Skills module in Command Center — a full evidence management system that replaces the inline JavaScript data in skills.html with a managed backend. This was a substantial scope:

**Backend:** 6 Supabase tables (`evidence_domains`, `evidence_skills`, `evidence_sources`, `evidence_skill_links`, `evidence_cert_details`, `evidence_item_details`) with full CRUD, RLS policies, and seed data migrated from the existing skills.html inline data. 25 API endpoints under `/api/evidence/`, plus a manifest generation endpoint that can publish the entire skills dataset as a static JSON file via the existing git handler.

**Frontend:** A 4-tab dashboard page at `/dashboard/add-skills` — Evidence Inventory (source listing with type badges and skill chips), Skill Mapper (interactive grid for skill-to-source linking), Taxonomy (domain and skill management), and Publish (manifest preview + git publish action).

**Pipeline hooks:** Fire-and-forget evidence source creation wired into both the MadLab prototype deploy and teardown publish pipelines. When a bot or teardown publishes, an evidence source record is auto-created and a notification suggests mapping skills in the dashboard.

The migration was executed through Supabase's SQL Editor via browser automation — Monaco editor injection with `window.monaco.editor.getEditors()[0].setValue(sql)` and keyboard shortcuts (Cmd+Enter) because the Run button clicks weren't registering. This is the third time we've executed Supabase migrations through this browser automation path, and it's becoming a reliable pattern despite its indirectness.

### The Mind the Gap Module

The third phase extended the Add Skills module with a new tab: "Mind the Gap" — a prioritized backlog of skills, certifications, and tools Kiran should learn, organized by 8 categories:

1. **Critical Gaps** (5 items) — Product analytics, experimentation, user research, pricing strategy, roadmap tooling
2. **Recognized Certs** (4) — Product School PMC, Pragmatic Institute, CSPO, DeepLearning.AI
3. **Persona Cred** (4) — System design, Figma fluency, full-stack demo, NN/g UX
4. **Domain Specialties** (6) — Growth PM, AI PM, Technical PM, Platform PM, Payments PM, Digital PM
5. **Adjacent Skills** (5) — Customer success, product marketing, behavioral economics, design thinking, advanced SQL
6. **Tool Proficiency** (6) — Amplitude, Mixpanel, Jira/Linear, Miro, PostHog, Looker
7. **Frameworks & Methods** (2) — JTBD, OKRs
8. **Horizon Expanders** (5) — Neurodesign, graphic design, data storytelling, product writing, platform economics

Each item carries: priority level, persona relevance array (which of Kiran's 6 target personas would be impressed), learning plan (provider, cost, time estimate, alternatives), and demonstration idea (how to prove the skill with a portfolio piece). Total: 37 seed items.

**Technical implementation:** New `evidence_gap_items` table with CHECK constraints on category/priority/status enums, TEXT[] arrays for persona_relevance and tags, 3 indexes, RLS with service_role policy. 6 new API endpoints (list with filters, stats aggregation, single item, create, update, delete). Frontend GapTab component (~280 lines) with stats summary cards, category filter pills, status filter row, grouped-by-category card list with expand/collapse for detailed views.

### The Dual Scroll Context Bug

The smoke testing revealed an interesting layout issue. When expanding a gap item card, the card's expanded details (learning plan, demonstration idea, status selector, persona relevance chips) rendered correctly in the DOM — confirmed via JavaScript inspection (298px height, visible, correct colors, text content present). But scrolling via mouse wheel produced a completely black viewport where the content should have been.

The root cause: the main content area has `overflow-y: auto` on the `<main>` element, and the page body also scrolls. Mouse wheel events at certain coordinates scrolled the window rather than the main content area, pushing the viewport into empty space beyond the content. The fix for testing was using `scroll_to` with element references rather than coordinate-based scrolling, which let the browser handle the scroll context correctly.

This is a pre-existing architectural issue in the Command Center layout, not specific to the GapTab. But it's worth noting because it cost ~15 minutes of diagnostic time during smoke testing — time spent injecting red backgrounds, checking viewport positions, and comparing bounding rects when the answer was simply "wrong scroll context."

### Migration Execution Across Three Context Windows

The session spanned three context windows due to the scope. The compaction boundaries fell at particularly inconvenient points: once while the Supabase SQL Editor Run button was failing to respond, and once while the GapTab frontend component was being built. The continuation prompts carried enough context to resume without loss, but the three-window span meant that design decisions made in window 1 had to be reconstructed from summaries in windows 2 and 3.

The seed data was inserted in 7 batches across the Supabase SQL Editor: DDL first (table + indexes + RLS), then critical gaps, recognized certs, persona cred, domain specialties, adjacent skills, tool proficiency, and finally frameworks + horizon expanders. All 37 items verified via SQL count query matching expected totals per category.

### What Shipped

Two complete modules in one day: the evidence management system (4 tabs, 25 endpoints, 6 tables) and the Mind the Gap learning backlog (1 tab, 6 endpoints, 1 table, 37 seed items). Both live in the Add Skills page at `/dashboard/add-skills`. The backend serves both through the same evidence router prefix. The frontend follows the same tab pattern, CSS variable theming, and card component architecture as the rest of Command Center.
