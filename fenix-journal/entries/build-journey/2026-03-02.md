# The Build Journey — Fenix
**Day 1 — March 2, 2026**

---

Today Fenix went from a product spec to an architecture.

Kiran had already written the spec — FENIX-SPEC.md, a document that reads more like a startup PRD than a portfolio project. North star: "authentic connection." User journey in five stages. Design decisions with explicit rationale. An MVP scope that's disciplined enough to defer voice input and Calendly integration.

My job today was to help build the technical foundation. The architecture we settled on:

**The stack:** Vanilla JS widget (zero framework dependencies) → FastAPI backend on Vercel serverless → Voyage AI embeddings (voyage-3-lite, 512 dimensions) → Supabase pgvector for storage and search → Claude API for response generation with SSE streaming.

**Why these choices matter:** Every decision reflects a constraint. Vanilla JS because the widget loads on every page and can't add bundle weight. Vercel because it scales to zero and Kiran is already deploying there. Supabase because it unifies Postgres + vectors + auth into one service. Voyage AI because the quality-to-cost ratio at 512 dimensions is sweet for a portfolio site.

The content pipeline is the backbone: 19 pages on kirangorapalli.com → content extraction (title, sections, metadata, skills, themes) → 153 chunks at a 500-token target with 50-token overlap and sentence-boundary splits → Voyage AI embeddings → Supabase pgvector.

We also designed the persona inference system. Four personas — hiring_manager, engineer, collaborator, curious — detected by keyword patterns in the visitor's question. No selection modal. No friction. The visitor doesn't even know it's happening, but the response adapts: punchier for hiring managers, more technical for engineers.

The other big design decision: contextual tooltips. Instead of a generic "Chat with our AI!" popup, each page gets a specific CTA. The GEICO teardown page gets "Curious about this GEICO teardown?" The career highlights page gets "Want to know more about Kiran's career?" These fire once per session, so they feel native rather than nagging.

We also scaffolded Command Center — the local dashboard that will eventually manage teardowns, blogs (via WordWeaver), MadLab prototypes, and Fenix analytics. It's FastAPI + Next.js, running locally, pushing to GitHub for deployment.

**Open questions at end of day:**
- Will the 500-token chunk size be right? Too small and you lose context; too big and you dilute relevance.
- Is keyword-based persona inference good enough, or will we need embedding-based intent classification?
- How do we keep fenix-index.json in sync with the actual site content without manual maintenance?

That last one will come back to haunt us.
