# The Build Journey — Fenix
**Feb 27 — 2026-02-27**

---

Today was architecture and infrastructure work disguised as portfolio polishing. It was the kind of day that sets up everything that comes next.

Kiran arrived with five interconnected tasks, but the throughline was clear: **prepare the site to be understood by both humans and AI.** For humans, that means explainer icons that teach. For AI (eventually Fenix), that means structured metadata and a semantic content graph.

The explainer icon work was the visible layer. We built a slide panel system that appears on teardown pages, prototype overviews, MadLab categories — anywhere someone might encounter an unfamiliar concept. The panel opens from the right edge (mobile bottom-sheet on small screens) and contains five structured sections:

- **What This Is** — Plain-English definition
- **Key Concepts** — Related terms explained simply (e.g., "LangGraph helps different AI agents talk to each other and make decisions")
- **Why It's Here** — Why this concept matters for understanding the work
- **What to Look For** — What to pay attention to when viewing the content
- **My Take** — First-person reflection on how Kiran approaches this concept
- **Connected To** — Cross-links to related content on the site

18 explainer icons were placed across 4 pages: GEICO teardown (4), Instagram teardown (4), Insurance chatbot overview (3), MadLab categories (7). The UI is accessibility-first: keyboard navigable, focus-trapped, screen-reader friendly, respects prefers-reduced-motion.

But the work didn't stop at implementation. Kiran made a deliberate choice: **every explainer needs to pass the same content checks as the portfolio itself.** No passive voice. No hedge language. No corporate softening. If the portfolio is going to teach, it teaches with the same clarity and directness as the rest of the site.

This meant running a full content audit on all 21 explainer blocks, finding violations (a few passive constructions, some smart quotes that needed escaping), and fixing them. It's the kind of quality control that separates "I put something up" from "I built this right."

The deeper architecture work was the Fenix Foundation layer. This is where it gets interesting.

**Fenix Foundation consists of three layers:**

1. **Fenix Index** — A single JSON file (fenix-index.json) at the site root that maps:
   - All 6 published content items with summaries
   - 5 hub pages and their relationships
   - Taxonomy of 14 skills (conversational AI, LangGraph, system design, etc.)
   - 10 themes (enterprise transformation, user adoption, product methodology, etc.)
   - 4 industries (fintech, insurance, banking, enterprise software)
   - 5 relationship types connecting everything

2. **JSON-LD Structured Data** — Added to 7 pages using schema.org:
   - Person schema on homepage
   - Article schema on blog posts and teardown pages
   - CollectionPage schema on hub pages
   - SoftwareApplication schema on prototype pages

   This serves dual purpose: SEO for humans, machine-readable semantics for AI.

3. **Data Attributes on Elements** — HTML elements tagged with:
   - `data-section-id`, `data-section-type`, `data-skills` on teardown sections
   - `data-project-id`, `data-project-status`, `data-project-category`, `data-project-skills`, `data-project-themes` on content cards
   - `data-explainer-type`, `data-explainer-context` on explainer panels

What this architecture enables: Fenix can eventually read the site and not just index it, but *reason about it.* It can understand that a blog post about AI adoption relates to a teardown about chatbot design, which connects to a prototype using LangGraph, which demonstrates skills in conversational AI and system architecture. It can surface relevant content based on what a visitor is asking about.

The key insight: **this is not over-engineering.** This is laying groundwork that makes the next layer of work possible without rework. Every line of metadata is something Fenix won't have to infer later.

By end of day:
- 18 explainer icons deployed across the site
- All explainer content audited and corrected
- fenix-index.json created and validated
- JSON-LD added to 7 pages
- Data attributes on content elements
- All changes validated locally (zero Netlify credit spent, as promised)

The architecture is now in place. The site is self-documenting. The content graph exists. When Fenix is built, it won't have to guess — it can read the metadata, understand the relationships, and reason about what matters.

**Status at end of day:** The portfolio site is no longer just a collection of pages. It's a structured knowledge base with semantic layers that humans can read (through explainers) and machines can reason about (through JSON-LD and data attributes). This is what "building for scale" actually means: infrastructure work that lets you move fast later because you did the thinking first.
