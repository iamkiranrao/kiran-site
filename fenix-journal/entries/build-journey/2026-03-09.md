# The Build Journey — Fenix
**Day 7 — March 9, 2026**

---

Today the Resume Customizer went from "broken in production" to "working correctly across all three templates." It was a deep dive into the XML internals of .docx files, the hidden structure of resume templates, and what happens when implicit assumptions about document structure collide with reality.

**The macOS temp directory problem.** The error `[Errno 13] Permission denied: '/tmp/command-center/jobs/36e7d720/Resume.docx'` looked like a one-off until Kiran started asking questions. Why does `/tmp` behave differently on macOS? Because it's a symlink to `/private/tmp` and sandboxed processes get different permissions depending on how they're launched. But more importantly: if this pattern is in one place, where else is it?

The answer: six places total. `resume_pipeline.py`, `teardown_service.py`, `madlab_service.py`, `job_central_service.py`, `wordweaver_service.py`, and the `wordweaver` router all hardcoded `/tmp/command-center/...` as the jobs directory. They all got converted to use `os.path.join(tempfile.gettempdir(), "command-center", ...)` which resolves correctly on any OS, regardless of how the process is launched. One problem fixed globally. That's maintenance hygiene.

**The resume template parsing bugs — the hard part.** Once permissions were fixed, the output was completely broken. Sections not rendering. Bullets appearing in the wrong places. Empty paragraphs. The quality check reported 7 errors and 22 warnings. This is where most people would start making targeted fixes. Kiran asked: what's *fundamentally* wrong with how bullets are being identified?

Three critical bugs emerged:

1. **Experience bullet matching destroyed company headers.** The code was looking for paragraphs where `company_name in text AND has_br`. But in the Detailed template, the `<w:br/>` elements live on the *header* paragraphs (like "EXPERIENCE[BR]VP OF PRODUCT | Avatour..."), not the bullet paragraphs below them. So it was replacing headers with bullet text. The fix: a new `find_bullet_paragraphs()` function that correctly locates the header first, then collects the bullets below it.

2. **First Republic and Wells Fargo AVP bullets were never touched.** Their headers didn't have `<w:br/>`, so they got zero matches. The new function handles both template styles.

3. **Skills reorder failed silently.** The code checked for `","` delimiters, but the template uses `"│"` (pipe). Also checked `not has_br`, but the skills paragraphs have `<w:br/>`. The reorder logic got updated to detect and handle both delimiters, and the pipeline got updated to match skills paragraphs regardless of BR presence.

4. **Summary overwrote contact info.** `find_summary()` was matching the first long paragraph (contact info with phone/email) instead of the actual summary. Now it looks for a "SUMMARY" header first, then falls back with better detection logic.

All three templates now parse correctly. The bullets go where they're supposed to go. The quality check passes.

**The real-world validation track.** While debugging the resume customizer on templates, Kiran was simultaneously running the tool on his own resume against an actual job description (Lead Digital Product Manager – Alerts & Messaging at Wells Fargo). This served multiple purposes:

1. **Validation.** If the tool doesn't work on Kiran's own resume, it's not ready. Period.
2. **Requirements clarification.** The tool is supposed to customize without changing bullet counts or page length. When he saw empty bullets in the output, he realized the *prompt* needed to enforce this, not just the code.
3. **Gap identification.** Scoring his resume against the JD revealed three main gaps: no email/SMS channel ownership stated, no enterprise messaging platform tools (Braze, Salesforce Marketing Cloud) in the skills section, and summary rendering issues from merged tracked changes.
4. **Iterative improvement.** Each gap identified became a specific change (add SMS to tagline, add Braze to tools, fix summary text). Each change got scored to measure impact. The final resume moved from 79/100 to 91+ on the match score.

The process was methodical: analyze → score → identify gaps → make changes → re-score → refine. Three iterations on the resume, each producing a cleaner version with yellow-highlighted changes so Kiran could review what changed and why.

**The prompt update.** The breakthrough on the bullet count issue was recognizing that the Claude prompt needed to be *told* exactly how many bullets each company should have. So Kiran added a bullet-count extraction step:

1. Parse the template and build a `bullet_map` for each role (e.g., {"Avatour": 3, "Wells Fargo Digital": 5, ...})
2. Pass this map to the Claude prompt in the context
3. Update the prompt to enforce: "Maintain exactly X bullets for [company name]"

This moves the constraint from "try to preserve the structure" to "you *must* output exactly this many bullets." Much stronger.

**Status at end of day:** All three resume templates (Detailed, 2-Pager, 1-Pager) parsing correctly. Bullet matching logic completely rewritten and tested against all three template variants. Resume customizer validated on Kiran's own resume with 87/100 match score against target JD. Tracked changes workflow working (Word document with strikethrough + underline annotations for review). Prompt updated to enforce exact bullet counts per company. All 6 hardcoded temp directory paths converted to use `tempfile.gettempdir()`. Pipeline ready for end-to-end testing.
