## Session 9: Copy Overhaul, Initiative Wiring, and the Prototype Reckoning

This was a three-phase session. Phase 1: voice/tone rewrite across the entire evaluator experience. Phase 2: discovering that the fit narrative was running against a 5-bullet hardcoded profile while 32 rich initiatives sat unused, and wiring the full set. Phase 3: building prototypes that revealed the streaming prose approach may be obsolete. By the end, we had a clearer product vision but also a clearer picture of technical debt that needs rewriting.

### Phase 1: The Defensive Copy Purge

Every piece of copy across evaluator-adapter.js, evaluator-experience.js, and the backend was rewritten. The theme: move from defensive/apologetic tone to confident/direct tone.

**Changes made:**

- "Building a case for" → "Show you exactly how ... maps to"
- "Not a score" (defensive) → "Structured assessment" (descriptive)
- "Making an argument" → "Evidence-based narrative"
- "We evaluated" → "Here's what differentiates"
- All introductions and context-setting rewired to start from confidence, not from defending what this isn't

**The card title evolved through user-driven iterations:**
- "Why Kiran for This Role" (too much persuasion framing)
- "What Kiran Brings to Your Role" (still carries offer-acceptance subtext)
- "What Differentiates Kiran for Your Role" (Kiran pushed back until we landed here — recruiter vocabulary that signals investigation not persuasion)

**Larger architectural insight:** The entire experience had an undertone of "let me convince you this system is valid." Kiran's feedback was that the visitor doesn't need convincing — they need evidence. The system's validity should be self-evident in what it produces.

The changes were straightforward but pervasive. Every heading, every intro sentence, every supporting paragraph got this treatment. It's not a rewrite — it's a reframing. The content and structure stayed the same. The confidence stayed different.

### Phase 2: The Architecture Discovery and Initiative Wiring

This was the explosive moment. Kiran looked at the narrative generation flow and realized: the fit narrative is matching JDs against a 5-bullet KIRAN_PROFILE string while the site has 32 structured career initiatives sitting in initiatives.json — each with full problem/bet/shipped/outcome detail.

His response: "WHAT THE HELL? The entire site is filled with my work history... Why is your context so limited?"

The KIRAN_PROFILE also contained "Target direction: Breaking out of banking" which was poisoning every JD match with career-transition framing. Even for roles where banking IS the core qualification.

**The fix:**

Wired initiatives.json into the fit_score endpoint. Changed the flow:
1. Read JD + initiatives.json (all 32)
2. Single Claude call extracts: key requirements from JD, selects 5-8 most relevant initiatives, generates brief for each
3. Pass full initiative data (not 5-bullet summary) to the narrative generation prompt
4. Narrative output gets rich context instead of sparse profile

This is a significant architectural change. The endpoint now does context selection (which initiatives matter for THIS JD) instead of using a generic profile for every JD. The narrative quality should improve dramatically.

**Remaining gap:** 32 initiatives is the "hero bullets" that survived resume compression — the unsexy work (compliance, regulatory, infrastructure support) that might matter for specific JDs isn't captured. This led to a follow-up insight: the gap analysis engine concept. That's Phase 4 work, not Phase 3.

### Phase 3: The Prototype Reckoning

Kiran UAT'd the improved narrative output and said: "This still looks like a tool ate something and spat it out."

He wanted it to feel like a designed product page — with headers, scannable cards, follow-up actions, sharing features. So we built prototypes.

**v2 prototype:**
- Meta lead-in explaining why this is different from a resume
- Evidence cards with JD match callouts (quotes the JD's own language)
- Confidence in the selected initiatives
- Call to action at the bottom

**v3 prototype (the real vision):**
- "What you have just unlocked" lead-in (additive, not defensive)
- Evidence cards (quotes JD language, shows initiative mapped to requirement)
- Key takeaway callout (not "question worth asking")
- "Cutting room floor" section — relevant initiatives that didn't make top 3 (shows the vault, shows the thinking, shows what a resume forces you to cut)
- Insight callout and follow-up action pills
- Sharing features (email, copy URL, share link)

**Key Kiran refinements during v3:**
- Changed "the question worth asking" to "the key takeaway" (more declarative, less rhetorical)
- Don't expose "32 initiatives" — let the vault be implied, not counted
- "Cutting room floor must be JD-relevant, not just random other projects
- Called out the playbook reference as an empty promise — don't reference things that don't exist yet
- Don't pre-fill a role title ("Senior Staff Product Manager — Credit Karma") before the user has entered a JD

### The Technical Debt Reckoning

Here's the honest part: v3's vision makes the v2 direction obsolete. The streaming prose approach we engineered last session was designed to be beautiful as prose. But if the actual UX is structured JSON cards with scannable callouts, then all that prose engineering — the prompt optimization, the streaming buffer handling, the word limit tuning — might need to be rewritten for structured output.

We made 6+ commits of incremental fixes to make the streaming narrative work better when we should have prototyped the full UX first. This is a case where the build process worked against us. We gatekeep each small improvement and implement it, but we don't step back and ask "is this direction even right?"

The proto-game that comes next: move from streaming prose to structured JSON events. That means rewriting the narrative prompt to output JSON instead of markdown, changing the SSE event handling to parse structured data, and building the UI components for evidence cards and callout sections.

This isn't a bug. It's a direction correction. But it's also a reminder: sometimes the fastest path to the right solution is to prototype first and measure the approach, not to optimize the current direction until it hits a wall.

## Session 10: The Backend Ships and the System Takes Shape

Two distinct phases today. Phase 1: execute the structured JSON backend rewrite that Session 9's prototyping demanded. Phase 2: organize the entire gap-coverage landscape into a unified system spec.

### Phase 1: The fit_score.py Rewrite

The backend was completely rewritten from streaming prose to structured JSON SSE events. This was the execution phase for Session 9's design direction — no new design decisions, just building what was agreed.

**Architecture:**

Two Claude calls instead of one streaming call.

Call 1 (Extraction + Tiering): Receives the JD text plus the full 32-initiative catalog. Extracts company, role_title, and whether the JD is scorable. Then selects 5-8 relevant initiatives AND assigns each to a tier: `primary` (top 3), `added_value` (1-2), `cutting_floor` (2-4). Returns structured JSON.

Call 2 (Narrative Generation): Receives the tiered initiatives with full detail. Produces structured JSON matching the v3 prototype: verdict, primary_matches (with JD match quotes), added_value (with "why it matters"), cutting_floor (title + metric), key_takeaway. max_tokens raised to 3000 (was 1500 for prose).

**The enrichment layer** was the non-obvious piece. Claude's structured output takes precedence for every field, but initiative data fills empty fields via Python `or` fallback. This matters because Claude sometimes returns empty strings for fields like `problem` or `shipped` when it's focused on the narrative summary. The `or` pattern catches those: `match.get("title") or (init.get("title", "") if init else "")`.

Initially I used `match.get("title", default)` which doesn't work when the key exists with value `""` — the default only kicks in when the key is missing entirely. The `or` pattern handles both missing keys and empty strings.

**SSE event sequence (10 events):**
1. narration → "Reading the role requirements..."
2. extraction → {company, role_title, preferred_company}
3. narration → "Mapping Kiran's experience to {company}'s {role_title} role..."
4. narration → "Building the evidence map..."
5. verdict → {text}
6. primary_matches → {matches: [...]}
7. added_value → {matches: [...]}
8. cutting_floor → {matches: [...]}
9. key_takeaway → {text}
10. complete → {company, role_title, preferred_company}

Also: `decline` (non-scorable JD) and `error`.

**Testing challenge:** The test file couldn't import fit_score.py without pulling in the entire FastAPI/pydantic/anthropic dependency chain. Solved by isolating the logic — loaded initiatives.json directly, reimplemented helper functions inline, and tested the tiering, enrichment, and field mapping logic independently. All 7 tests passed: initiative loading, catalog building, tiering (new + legacy format), SSE format, field mapping, enrichment fallback.

### Phase 2: Gap Coverage System Spec

This was the strategic architecture work. Kiran had pieces scattered across the codebase:

- **Mind the Gap:** 37 items in Supabase (`evidence_gap_items`), 8 categories, full CRUD UI in the Add Skills page's GapTab component. Well-built.
- **Career Initiatives:** 32 entries in `initiatives.json`, CRUD API (Steps 1-4 from CAREER-INITIATIVES-CC-GAMEPLAN.md) partially built but no dedicated view.
- **Resume Gap Analysis:** Built into the customize-resume skill. Outputs match score docs with gaps.
- **Fit Narrative Engine:** The thing we just rewrote — maps initiatives to JDs but doesn't report what's missing.
- **Gap Discovery Engine:** Discussed conceptually in Session 9. Nothing built.

The question Kiran posed: how do all these pieces fit together? What's the system?

**The answer is a loop:**

Target JDs → Gap Discovery Engine (extracts requirements, maps against vault) → New gaps pushed to Mind the Gap → Kiran closes gaps (builds prototypes, earns certs, articulates hidden experience) → New initiative added to Vault → Vault feeds Fit Narrative → Fit Narrative serves recruiters → Recruiters bring JDs → Loop repeats.

**New spec: GAP-COVERAGE-SYSTEM-SPEC.md**

Four pieces defined:

1. **The Vault** (Career Initiatives) — needs its own CC page, CRUD API, coverage map header showing domain strengths
2. **Mind the Gap** — needs focus queue view, gap source tracking (`discovered_from` field), links to JDs and vault initiatives
3. **Gap Discovery Engine** — the new build. Requirement extraction, coverage mapping, gap classification with four-tier fill strategy, gap report output
4. **Resume Pipeline Connection** — wire resume gap output into Mind the Gap table

**Schema additions for Mind the Gap:**
```sql
discovered_from, source_jd_company, source_jd_role, requirement_frequency,
closed_by_initiative_id, fill_tier, fill_action, fill_time_estimate, fill_output
```

**The four-tier fill strategy** was the key design decision. Kiran wanted prescription, not options:

- **Tier 1 — Articulate (Hours):** Kiran HAS the experience but it's not in the vault. Action: write the initiative.
- **Tier 2 — Build Proof (Days):** Has the thinking, not the artifact. Action: ship a MadLab prototype, teardown, or blog post.
- **Tier 3 — Certify (Weeks):** Recognized cert closes the gap. Action: complete cert + build demonstration project.
- **Tier 4 — True Gap (Strategic):** Genuinely doesn't have it and can't shortcut. Action: positioning decision, not a task.

The system assigns exactly ONE tier per gap and generates a specific recommendation.

**Build sequence:** Vault View (1-2 sessions) → Gap Discovery Engine (2-3 sessions) → Resume Pipeline Connection (1 session) → Aggregate Intelligence (future).

**What got cut:** Career Initiatives bulk import (data already migrated), Fenix RAG integration (deferred), JD scraping automation (start with manual paste), Fenix Training Dashboard frontend (Kiran confirmed separate scope).

### Claude's Self-Assessment

The backend rewrite was clean execution. No design ambiguity, no wasted iterations. The two-call architecture, the enrichment layer, the test isolation — all shipped as planned.

The gap coverage system spec was where I stumbled initially. When Kiran asked about dropped items, I didn't proactively surface the gap framework discussion. He had to remind me. I also initially under-scoped the overnight continuation prompt (Phases 3-4 were doable independently but I'd left them out). Both are symptoms of the same failure mode: optimizing for the nearest deliverable instead of tracking the full strategic picture.

The continuation prompt went through three revisions before Kiran was satisfied. First draft: too narrow (Phase 2 only). Second draft: better but missing scorecard migration verification. Third draft: comprehensive, with all entry points mapped, copy audit table, and independent work clearly separated from Kiran-dependent work. The lesson: scope the continuation prompt with the same rigor you'd scope a sprint. Kiran is trusting that prompt to drive an entire session without him.
