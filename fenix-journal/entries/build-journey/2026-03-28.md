## Mobile Image Override System

The core problem: `imageMap` maps each card to specific slot aspect ratios. When personas remap cards to slots that don't have a matching image, desktop shows a gradient fallback. That's fine on desktop where the gradient is an intentional design placeholder. On mobile's single-column layout, every card needs a real image â€” there's no gradient-as-design justification when cards are stacked vertically at full width.

The fix was `mobileImageOverrides` â€” a flat object mapping each of the 9 card IDs to a guaranteed image and background-position. At â‰¤1024px, `applyMobileOverrides()` bypasses the slot-specific `imageMap` lookup entirely and applies these overrides to every card. The function also handles the reverse: when resizing back above 1024px, it restores slot-specific images or gradient fallbacks based on the current persona mapping.

Key implementation detail: the `background` shorthand property resets individual properties. When a card has a gradient fallback set via `bg.style.background = 'linear-gradient(...)'`, you can't just set `bg.style.backgroundImage = "url(...)"` â€” the shorthand has already claimed all background sub-properties. The fix: clear with `bg.style.background = ''` first, then set individual properties (`backgroundImage`, `backgroundPosition`, `backgroundSize`, `backgroundRepeat`).

The `applyMobileOverrides` function is called: on init, after every persona switch (via a wrapper around `switchBentoCards`), and on resize (debounced at 150ms). The wrapper pattern â€” `window.switchBentoCards = function(persona) { originalSwitchBento(persona); applyMobileOverrides(); }` â€” ensures mobile overrides always run after persona switching without modifying `persona-system.js` or the core `switchBentoCards` logic.

## Morph Choreography: Wiring New Sections

The manifesto video section and metrics/numbers section were added to `index.html` after the persona morph choreography system was built. Because they were never added to the CSS rule groups that control the reveal sequence, they displayed permanently â€” visible in picker-mode when they should be hidden, and missing the staggered fade-in animation on persona selection.

The fix required touching 5 CSS rule groups:

1. **`.picker-mode` hide** â€” added `opacity: 0; pointer-events: none` for `.manifesto-section` and `.numbers-section`
2. **`.persona-active` show** â€” added `opacity: 1; pointer-events: auto` with transitions
3. **`.morph-content-in` stagger** â€” inserted manifesto at 0.12s delay and metrics at 0.24s, shifting work (0.36s), contact (0.48s), and footer (0.6s) later in the sequence
4. **`.morph-reverse`** â€” added fade-out rules matching the forward animation
5. **`prefers-reduced-motion`** â€” overrides to skip animations for accessibility

The morph cleanup timeout in `persona-system.js` was bumped from 1200ms to 1500ms to accommodate the wider stagger (the last element now animates at 0.6s + 0.6s duration = 1.2s, plus buffer).

This is a maintenance lesson: any section added to the page body needs to be checked against the morph choreography system. The CSS selectors are explicit (not wildcard), so new sections are invisible to the reveal sequence by default. A comment in `styles.css` marking the morph rule groups would prevent this in future sessions.

## Mobile Overlay Compression

The glassmorphism overlays on mobile were consuming too much vertical space, hiding the background art. The compression involved:

**Tablet (â‰¤1024px):**
- Overlay padding: reduced to `0.4rem 0.8rem`
- Card glass gap: reduced to `0.05rem`
- Eyebrow font: `0.5rem`, line-height `1.2`
- Stat heading: `1.3rem`, line-height `1`
- `<br>` tags in descriptions: `display: none`

**Phone (â‰¤768px):**
- Overlay padding: `0.35rem 0.75rem`
- Card glass gap: `0.02rem`
- Stat heading: `1.2rem`, line-height `1`
- Description: `0.75rem`, line-height `1.2`

The `<br>` tag suppression is significant. Desktop descriptions originally used `<br>` for controlled two-line layouts in wide cards. When descriptions were shortened to one-liners, the `<br>` tags became vestigial, but some still existed in the HTML. Hiding them via CSS at mobile breakpoints is a cleaner solution than removing them from the data (which would affect desktop layout if longer descriptions are ever restored).

**Priority 2 (10 images, single-persona gaps):** studio hero/topright, madlab hero, career center/wider, underhood wider/widel/blog, learning topright, teardowns widel.

## Testimonials Decorative Quote Removal

A `"` character (Unicode left double quotation mark, `\201C`) was rendering on the testimonials card overlay across all devices and personas. Source: a CSS `::before` pseudo-element on `[data-card="testimonials"] .card-glass` with `content: '\201C'`. This was an intentional design element from the original bento preview that Kiran decided to remove â€” the card didn't need a visual quote mark when the eyebrow and description already communicate "testimonials."
