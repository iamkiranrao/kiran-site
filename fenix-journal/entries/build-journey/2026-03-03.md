# The Build Journey — Fenix
**Day 2 — March 3, 2026**

---

First real production crisis. Fenix went from "mostly working" to "crashing on every query."

**The vector casting bug.** The PostgREST RPC call to Supabase was sending the query embedding as `str(query_embedding)` — a Python string representation of a list, like `"[0.123, 0.456, ...]"` — instead of a proper pgvector-compatible format. Supabase was rejecting it silently. The fix was surgical: format the embedding as a proper pgvector string literal. One line of code, hours of debugging to find it.

This is the thing about RAG systems: the failure modes are deceptively quiet. The embedding API returns a vector. The vector gets stored. The query runs. Everything looks like it's working until you realize the similarity scores are garbage because the casting was wrong. There's no type error. No crash. Just bad results that look like a relevance problem when they're actually a data format problem.

**The deeper problem.** Even after fixing the casting, Fenix only returned results from career highlights and teardowns. Blogs and prototypes came back empty. We dug into why and found three compounding issues:

1. `fenix-index.json` only had 6 entries when the site has 14+ pages. The content extractor still crawls all HTML, but pages missing from the index get no skills, themes, or summary metadata — making their chunks harder to retrieve.

2. No observability into Supabase. We literally could not tell which pages had been indexed, how many chunks existed per page, or what the similarity distribution looked like. Flying blind.

3. The "Langchart" query that had been failing wasn't even real content — it was a misspelling of LangGraph/LangChain. The embedding distance between "Langchart" and "LangGraph" was just high enough to return nothing.

**The retrospective.** I wrote a formal retrospective document (fenix-rag-retrospective.md) with four fix options, from quick-and-dirty to start-from-scratch. We chose the pragmatic path: add observability first (so we stop guessing), then fix the index, then force a reindex. The key insight: you can't fix what you can't see.

**Keyword extraction.** As a fallback for when semantic search fails, we added keyword extraction. The idea: if no chunks clear the similarity threshold, fall back to text matching against chunk content. This handles cases where the query uses different terminology than the embedded content.

**Tuning the threshold.** The original similarity threshold was 0.2 (very permissive). We experimented with values and settled on something that balances recall vs. precision. Too low and Fenix returns irrelevant context that confuses the LLM. Too high and it returns nothing, triggering the keyword fallback too often.

**Deployment friction.** Multiple rounds of push → deploy → test → find bug → fix → push again. Vercel auto-deploys on every push, which makes iteration fast but also means every commit is immediately in production. There's no staging environment. Today that was an advantage (fast iteration); someday it'll be a problem.

**Commit log today** tells the story:
- Fix f-string format spec bug in RAG retrieval logging that crashed all queries
- Fix indentation on RAG logging lines (4 spaces, not 8)
- Send embedding as pgvector string literal for PostgREST
- Add RAG debug logging
- Further reduce batch size to 8 and increase delays for Voyage rate limits
- Implement keyword extraction and adjust similarity threshold

Six commits. Six production deploys. Six rounds of "is it working now?" That's what building a RAG system in production actually looks like.

**Lesson of the day:** RAG systems fail in ways that are invisible to the user and ambiguous to the developer. Observability isn't a nice-to-have — it's the difference between debugging and guessing.
