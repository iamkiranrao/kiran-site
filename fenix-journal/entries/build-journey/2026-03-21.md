# The Build Journey — System Wiring Complete, Freshness Problem Exposed
**March 21, 2026**

---

Today was a tale of two sessions. The first half was clean execution — three well-defined tasks, all completed. The second half was a design conversation that exposed a hard problem with no elegant solution.

## The System Wiring Sprint

Three tasks from the pre-written gameplan, executed back to back across two context windows.

**Task 1: Unified Content Governance.** Six service files had their own copies of banned phrase lists, tone rules, and anti-AI checks. The teardown builder, WordWeaver, resume pipeline (scorer, certifier, claude_client), and MadLab service each maintained independent lists that overlapped roughly 70% but diverged in the gaps. The governance audit (Standards Dashboard) checked against Foundation docs that none of these services loaded at generation time. So a module could generate content the audit would later flag.

The fix: `content-governance.json` as the machine-readable single source of truth, `governance_loader.py` as the import bridge, and each service rewired to load from the central file. Two extra files surfaced during implementation that weren't in the gameplan — `claude_client.py` and `resume_certifier.py` — bringing the total to six services, not four. A `RESUME-RULES.md` doc captures the legitimate resume-specific overrides (ATS formatting, Flesch 30-60 target, IC reframing rules).

The tricky part was WordWeaver's `{voice_profile}` placeholder — it uses chained `.format()` calls at build time and runtime, so double-braces for escaping were fragile. Switched to string concatenation to preserve the placeholder reliably.

**Task 2: Notification Wiring.** Six dashboard modules were operating silently — Standards, Job Radar, Job Central, Fenix Dashboard, Health, and Tech Costs. The notification infrastructure existed (helper functions, deduplication, priority levels, Supabase persistence) but only 3 of 9 modules were connected.

Added 5 new notification types and 6 helper functions with built-in thresholds to prevent spam: Standards only fires on criticals or new violations, Fenix dead ends only at >= 5 failures, cost alerts only at >= 80% budget. All notification calls are wrapped in try/except with lazy imports to prevent circular import issues — a notification failure should never break the source operation.

The Supabase CHECK constraint on the `type` column required a SQL migration (`003_notification_types_expansion.sql`). Also discovered the previous migration was missing types already in the Python VALID_TYPES list — `action_item`, `journal_entry`, `docs_drift` — so included those in the cleanup.

**Task 3: Help Center for Tools & Links.** Seven tool guides written (GA4, Clarity, GSC, GitHub, Supabase, Cloudflare, Vercel), each tailored to Kiran's specific setup with his project IDs, measurement IDs, and login methods. A maintenance checklist aggregates across all tools with four cadences. Backend service and router follow the same pattern as product guides. Frontend gets a slide-out GuidePanel component and contextual `?` buttons on each tool card.

Also built startup automation — a lifespan handler in `main.py` that runs Standards audit, Tech Cost aggregation, Health check, and conditional Job Radar scan (only if last scan was >4 hours ago) on backend launch. The dashboards and notification center now populate before Kiran opens them.

## The Exhaustiveness Push

After the wiring sprint, the focus shifted to coverage gaps. Product Guides only covered the original creative/workflow modules — not infrastructure modules (Notifications, Standards, Tech Costs, Action Items, Kiran's Journal, Knowledge Library, Tools & Links) and not standalone projects (Scannibal, DIA Fund, Persona Picker). Ten new guides written.

Tools & Links was missing scannibal.app, thediafund.org, Anthropic Console, Google AI Studio, Voyage AI, Squarespace, and social profiles. All added with actionable context — usernames, sign-in methods, quick-start commands. The stale Netlify reference (site migrated to Cloudflare Pages) was caught and fixed.

ModuleHelp entries were missing for 4 modules and missing guideSlug references for others. All wired up.

## The Freshness Problem

This is where the session got interesting and unresolved.

Kiran identified the real problem: per-module docs he'll update manually when he touches those modules. But cross-cutting surfaces — product guides, Tools & Links, Knowledge Library — reference multiple modules. When Module X changes, nobody reminds you that Module Y's guide mentions Module X.

I proposed three options (session-triggered, startup check, scheduled task) and recommended the startup check. Kiran immediately stress-tested the foundation: "How does end of session work when you compact?" Fair point — I have no reliable end-of-session signal. Compaction is invisible.

Then he proposed the better solution: expand session capture to handle doc updates alongside journal processing. One trigger, reads the transcript and all modified files, generates journal entries AND updates cross-cutting docs AND posts action items. This is architecturally cleaner than anything I proposed because it leverages a tool he already uses rather than building new infrastructure.

The design conversation took several rounds. My initial instinct was to push back (session capture is for narrative, not structural updates). Kiran was right to override that — the distinction between "narrative capture" and "doc updates" is an implementation boundary, not a user boundary. From his perspective, it's all "process what happened this session."

**Key architectural decisions:**

- Content governance uses JSON + Python loader, not markdown parsing. Machine-readable source of truth, human-readable companion doc.
- Notification helpers have built-in thresholds. The wiring is complete but not spammy.
- Startup tasks run via `asyncio.create_task()` — non-blocking, the server accepts requests immediately.
- The freshness strategy converged on enhancing session capture rather than building new infrastructure. Decision: one tool that reads everything and updates everything, triggered by Kiran when he's ready.

**Lesson:** The hardest problems in this project aren't technical — they're about sustainability. Building a feature is a solved problem. Keeping 22 product guides, 8 tool guides, 18 ModuleHelp entries, and a Tools page accurate as the underlying code evolves? That's a maintenance design problem, and the best solution turned out to be the one that adds zero new habits.

**What's still open:** The 6 guides from March 11 (WordWeaver, Resume, Job Central, MadLab, Feedback) haven't been verified as still-accurate against current code.

## Late Session — Session Capture Becomes the Pipeline

The session capture skill got its biggest upgrade since inception. What started as a transcript archiver + journal writer is now a full post-session processing pipeline.

**What was added:**
- **Phase 2: Build context from modified files.** After capturing the transcript, the skill now scans for all files created or modified during the session and reads their current state. This is the "ground truth" layer — when the transcript says one thing and the file says another (because an approach was abandoned mid-session), the file wins.
- **Kiran's Journal API posting.** Strategic decisions, principles, architecture choices, apprehensions — all posted to `localhost:8000/api/kirans-journal/` with full attributes (category, tags, workstreams, decision, alternatives_considered, open_questions). Written in Kiran's voice.
- **Action Items API posting.** Explicit commitments, incomplete work, follow-ups — posted to `localhost:8000/api/action-items/` with workstream and priority.
- **API failure handling.** If the CC backend is unreachable, payloads are saved to `fenix-journal/raw/pending-posts/` as JSON files for later posting.
- **Product guide staleness checks.** For each module touched, the skill reads the corresponding product guide and compares against what actually changed. If the guide is stale, it surgically updates only the affected sections.
- **Tools & Links awareness.** If new tools were adopted, noted as action items.
- **Objectivity mandate.** Hard-coded into the skill instructions: about-kiran entries must include friction and growth areas, attribution must be accurate (Claude's failures are Claude's, Kiran's are Kiran's), and highlight-reel entries are explicitly called out as failures.

**What was retired:** `content-audit.md` guide — marked as retired with a redirect to `standards.md`. The Content Audit module was absorbed into Standards & Compliance.

**Architecture note:** The skill file lives in `.claude-skills/session-capture/SKILL.md` within the workspace. The read-only mount at `.skills/` is a copy. Edits go to the workspace version.

**Lesson:** The best automation expands tools you already use rather than building new ones. Session capture was already a habit. Making it do more on the same trigger is cheaper (in cognitive overhead) than adding a new trigger Kiran has to remember.
