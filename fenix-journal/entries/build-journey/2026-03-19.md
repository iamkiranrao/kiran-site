# The Build Journey — The Portrait Pivot
**March 19, 2026**

---

The persona picker underwent a fundamental creative direction change today. Not a CSS tweak or a color swap — a structural pivot from landscape cards with split character/text layout to portrait cards with full-bleed character art and glassmorphic text overlays. The old prototype still works but represents a dead branch. The new direction is architecturally different.

**The margin research that started it all.** The session opened with a seemingly small question: is the 60px side padding on the card grid optimal? Research across premium sites (Linear, Raycast, Vercel, Notion) showed that 24-40px is the industry standard for desktop side margins at 1200-1440px containers. Linear uses max-width: 1436px with 32px padding. Raycast uses max-width: 1204px with 32px. The persona picker was set at max-width: 1320px with 60px — wider margins than any reference site. The fix: 1440px max-width with 40px padding. Cards went from ~384px to ~437px wide. A 14% increase in card real estate from changing two CSS values.

**The color system overhaul.** Programmatic analysis revealed the gradient backgrounds had critical overlap. The three warm-tone personas (Seeker at H:39°, Explorer at H:12°, Inner Circle at H:36°) had a total hue span of only 28° — with the closest pair (Seeker ↔ Inner Circle) at just 4° apart. The accent colors were better separated but the gradients were visually indistinguishable.

The fix went through multiple iterations. The final locked palette uses six truly distinct color families: Steel Blue (#7B9ACC), Warm Gunmetal (#8A8580), Jade Green (#4DAF8B), Lavender Purple (#A07ED4), Cherry Red (#cb5c72), and Rusty Orange (#cb6923). The minimum accent pair distance went from 8 to 61. Each color is immediately nameable by a child — blue, gray, green, purple, red, orange.

Interesting lesson: I kept shying away from bold warm colors, defaulting to muted dusty tones (dusty mauve, sandstone gray). Kiran called it out: "Why are you shying away from distinct colors like red, yellows or oranges?" The Evaluator gets a committed blue, the Practitioner gets a rich green — but I was giving the warm personas washed-out variants. The lesson: if you're going to use a color family, commit to it.

**Typography and contrast fixes.** WCAG analysis caught several issues. The card title at `rgba(255,255,255,0.50)` failed AA at 3.8-4.2:1 across all cards. The card hook at 0.55 opacity was borderline. The card tag at 10px was below recommended minimums. Fixes: title and hook opacity bumped to 0.62, tag font size from 10px to 11px, Explorer accent lightened from #CC6B5A to pass AA. All text now passes WCAG AA across every card background.

The `image-rendering: crisp-edges` and `image-rendering: -webkit-optimize-contrast` filters from the v2 build were also removed. These were intended for pixel art and were making the Midjourney painterly images look harsh with visible aliasing. The browser's default bilinear scaling produces much better results at the 2048px → ~273px downsample.

**The naming pivot.** All persona names changed from conventional wordplay (Hunter Voss, Quest Parekh) to Bart Simpson-style double-meaning names: Merritt Hunter (merit hunter), Phil Thevoid (fill the void), Drew Skematics (drew schematics), Paige Turner (page-turner), Wanda Ring (wandering). The Inner Circle's Keshav Shivdasani remains the only real name — fitting, as it's the only persona that represents an actual personal relationship.

Titles dropped fictional companies entirely. Instead of "Talent Lead at FitCheck," the Evaluator now says "Recruiter · Hiring Manager." The title became a self-identifier mirror — the visitor should see their own role, not a fictional character's job. The hooks were also rewritten to match: each one is now in the visitor's own inner monologue, not a character's catchphrase.

**The portrait + glassmorphism pivot.** The original design spec actually called for 3:4 portrait cards (380-420px wide, 450-520px tall). Somewhere during the v2/v3 build, this shifted to 16:10 landscape because of concern about vertical scroll. But the fundamental problem with landscape was always there: people are vertical, the cards were horizontal, and the character art was fighting for space in a 42% side panel.

The new direction: portrait 3:4 cards where the Midjourney character fills the entire card. A glassmorphic text panel sits at the bottom — `backdrop-filter: blur(14px)`, `background: rgba(10, 8, 6, 0.45)`, thin accent-colored top border on hover. The text stack simplifies to: Title (accent color, small caps) → Name (Playfair Display, white) at rest, with the Hook sliding in on hover.

The hero image also pivots: Ken Burns CSS animation replaced with a Runway Gen-4 cinemagraph — dog breathing, steam rising, clouds drifting, light shifting. Delivered as a looping `<video>` element with the still image as poster frame fallback.

**Character re-specification.** All six characters are being regenerated at `--ar 3:4` with dramatically different compositions. Each character now has a unique: ethnicity (Black woman, Latino man, East Asian woman, South Asian woman, mixed-race non-binary, South Asian man), pose (seated-reclined, standing-forward, standing-angled, leaning-on-desk, casual-lean, seated-open), hand position, eye line, clothing style, and distinguishing prop. Persona-colored accent lighting is baked into each Midjourney prompt as a rim/fill light — steel blue catching Merritt's reading glasses, jade green on Drew's turtleneck collar, cherry red warming Wanda's denim jacket.

**Status:** The v4 Midjourney prompts are written and saved. The production portrait prototype can be built in parallel with Midjourney re-generation using the existing square images as placeholders. Key parallel workstreams: hero video element architecture, glassmorphic panel with title integration, persona-tinted bottom gradients, responsive breakpoints for portrait cards, backdrop-filter fallback for older browsers.

## Evening Session — From 44MB to 600KB

The second build session completed Phases 1 through 4 in a single sitting. All six Midjourney characters are generated, compressed, and live.

**The Midjourney style consistency pipeline.** The biggest technical challenge was getting all six characters to look like they belong in the same animated movie. The solution evolved through trial and error: Phil Thevoid became the "style anchor" — his Pixar-cartoon aesthetic was the strongest and most distinctive. Every subsequent character was generated with `--sref <Phil URL>` to reference his style. But `--sref` doesn't just copy style; it copies composition. Phil's image was a tight headshot, so every character rendered as a tight headshot.

The fix required two simultaneous adjustments: `--sw 30` (style weight reduced from default 100 to 30, loosening the sref's grip on composition) and front-loading framing language at the very start of the prompt: "Full upper body portrait of a character from waist up, camera pulled back to show torso and hands, wide medium shot." Midjourney processes prompts roughly front-to-back, so framing instructions must come before character descriptions to win priority. This is a transferable lesson for any Midjourney workflow using `--sref`.

**The `--cref` experiment.** For Keshav (Inner Circle), Kiran used a real friend's photo with `--cref` (character reference) alongside `--sref` (style reference). The combination at `--cw 65` struck the right balance — recognizable facial features (jawline, salt-and-pepper beard, grey temples) stylized into the Pixar aesthetic. This is a powerful technique for anyone building character-based designs that need to bridge fiction and reality.

**Image optimization — the WebP win.** The six Midjourney outputs arrived at 1856×2464 pixels, 7-8MB each as PNG. Total: 44MB for the character images alone. The optimization pipeline:

1. Resize to 600px wide (2x retina for ~300px desktop cards) — PNGs dropped to ~700KB each
2. PNG compression plateau — ImageMagick couldn't push below 640-768KB per image
3. WebP conversion at quality 85 — collapsed to 52-76KB per image. A 90% reduction over the resized PNGs
4. Hero image: 3360×1440 PNG (5.3MB) → 1920px wide WebP at quality 85 → 188KB

The HTML now uses `<picture>` elements with `<source srcset="*.webp" type="image/webp">` and `<img src="*.png">` fallback. Modern browsers get 384KB total for all six cards. Safari pre-16 and IE fall back to the PNG versions at ~4MB total. The three-tier fallback chain is: WebP → v4 PNG → v3 PNG (via `onerror`).

**Final page weight: 600KB** for a modern browser (28KB HTML + 188KB hero WebP + 384KB card WebPs). Down from an estimated 44MB+ with the original uncompressed assets. That's a 98.6% reduction.

**WCAG contrast audit passed.** All six accent colors pass AA against the near-black glass background (rgba(10,8,6,0.45)):

| Persona | Accent | Ratio | Status |
|---|---|---|---|
| Evaluator | #7B9ACC | 7.0:1 | AA |
| Seeker | #8A8580 | 5.5:1 | AA |
| Practitioner | #4DAF8B | 7.4:1 | AA |
| Learner | #A07ED4 | 6.1:1 | AA |
| Explorer | #cb5c72 | 5.1:1 | AA |
| Inner Circle | #cb6923 | 5.3:1 | AA |

Card name (#fff) at 20.0:1 and card hook (~rgba(255,255,255,0.65)) at 8.2:1 are well above threshold.

**Production swap complete.** `persona-picker.html` now serves the v4 production version. The old version is backed up as `persona-picker-v3-backup.html`. Keyboard accessibility (tabindex, role="button", aria-label, focus-visible, Enter/Space handler), `prefers-reduced-motion`, and glassmorphism `@supports not` fallback are all in place.

**What remains:** Live browser QA (hover animations, card click → localStorage, glassmorphic blur rendering), responsive breakpoint verification, and the hero video slot whenever the Runway Gen-4 cinemagraph is ready. The video element architecture is already in the HTML, commented out, waiting for `hero-loop.mp4`.

## Late Night Session — Documentation as Infrastructure

Tonight's session had zero CSS changes and zero JavaScript. Instead, it was an infrastructure project of a different kind: rebuilding the project's entire documentation system from scratch — twice.

**Round 1: The 7-doc consolidation.** The project had accumulated 41 markdown files at root level, many contradicting each other. The session from earlier today had already drafted 7 "super docs" (ARCHITECTURE, STANDARDS, PERSONA-PICKER, FENIX, COMMAND-CENTER, PLATFORM-MIGRATION, SITE-CONTENT) by reading every source file, validating against the actual codebase, and merging. A validation pass caught real drift: 3 persona hooks that didn't match the live HTML, font sizes documented as 28px when the CSS said 21px, a background color off by a full hex value, animation timings recorded as 500ms when they'd been tuned to 2.2s. All corrected.

32 files moved to `archive/`. Clean root.

**Round 2: The 22-doc restructure.** Kiran evaluated the 7-doc structure and decided it was still too coarse for how he works. A single COMMAND-CENTER.md bundling 18 routers meant that working on WordWeaver required loading context about Resume Pipeline and Fenix Training — irrelevant noise. A single SITE-CONTENT.md covering 25+ HTML pages meant the teardown section context bled into the blog section.

The new taxonomy:

- **3 foundation docs** (cross-cutting): ARCHITECTURE.md, CONTENT-STANDARDS.md, VISUAL-STANDARDS.md
- **2 product docs** (standalone initiatives): SCANNIBAL.md, DIA-FUND.md
- **7 CC module docs**: CC-RESUME-PIPELINE (resume + job central + job radar grouped because they share 10 services), CC-WORDWEAVER, CC-TEARDOWNS, CC-FENIX-JOURNAL, CC-FENIX-TRAINING, CC-AUDITING, CC-CORE
- **7 site page docs**: SITE-HOMEPAGE, SITE-TEARDOWNS, SITE-BLOG, SITE-MADLAB, SITE-CAREER, SITE-STUDIO, SITE-SUPPORT (lighter pages bundled)
- **3 kept as-is**: PERSONA-PICKER, FENIX, PLATFORM-MIGRATION

The key architectural decision was the "delta to world class" sections in each foundation doc. Rather than documenting only current state, each doc now includes a gap analysis against best-in-class references (Stripe's API guide for architecture, Apple HIG for visual, Mailchimp/GOV.UK for content) with quick wins, medium-term improvements, and long-term aspirations. This means the docs serve dual purpose: they're both operational references and roadmaps.

**The PROMPTGUIDE pattern.** 22 docs creates a discovery problem. The solution: a root-level PROMPTGUIDE.md with copy-paste-ready prompt blocks for every work area. Each block tells a new session which foundation doc(s) to read first, then which module doc, includes a 3-line status summary, and ends with a specific first action. It's an onboarding flow for AI sessions — the same concept as a developer's "getting started" guide, but designed for context-window efficiency.

**Cross-reference validation.** After the restructure, 15 files had broken cross-references — pointing to the old STANDARDS.md, COMMAND-CENTER.md, or SITE-CONTENT.md that had been archived or split. An automated scan caught them all: SITE-BLOG.md alone had 7 broken refs. All fixed to point to the current doc names.

**Technical lessons:**
- Documentation consolidation is essentially a database normalization problem. Round 1 was denormalization (merge everything). Round 2 was proper 3NF (separate concerns, eliminate transitive dependencies, keep foreign keys clean).
- Cross-reference integrity needs to be checked after any restructure — it's the documentation equivalent of running your test suite after a refactor.
- The "delta to world class" pattern is powerful. It turns a static reference doc into a living roadmap without cluttering the operational sections.

**Final counts:** 22 active docs in `docs/`, 35 archived files in `archive/`, 1 routing guide at root. Every doc has a continuation prompt. Every cross-reference validated.
