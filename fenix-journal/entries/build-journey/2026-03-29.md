**March 29, 2026**

## Action Item Reconciliation at Scale

The session opened with a systematic cross-reference of 70 Command Center action items against the live codebase. The approach: fetch all items from the CC API, then verify each one by reading the actual files. This revealed 20+ items still marked "todo" that were already implemented — a significant drift between the tracking system and reality.

Key technical discovery: the CC API uses PUT (not PATCH) for updates. PATCH requests returned 307 redirects and then "Method Not Allowed" without a trailing slash. This is a Django/FastAPI routing behavior that wasn't documented anywhere. Once identified, bulk updates went smoothly — 23 items marked completed in one batch.

The drift wasn't random. Most of the stale items were from the bento grid and persona system workstreams — areas where rapid iteration over multiple sessions meant features shipped faster than tracking could keep up. The observation generalizes: any workstream with high session velocity will have tracking drift. The CC API needs either automated status checks or a recurring reconciliation cadence.

The responsive CSS audit document specified 34 rules across three categories: Category A (slot-level sizing and min-heights by breakpoint), Category B (card-level typography floors and overlay compression), and Category C (card-in-slot targeted overrides for 8 specific persona×card×slot combinations). ## The Triptych: From Concept to Production

The biggest build of the session: replacing the two-column About section (intro paragraph + divider + 6-tile competencies grid) with a three-column "triptych" layout.

### Architecture

The triptych uses CSS Grid with `grid-template-columns: 1fr 1fr 1.2fr`. The third column is slightly wider (1.2fr) to accommodate the carousel's peek effect — showing roughly 18% of the next card requires slightly more column width to feel balanced.

**Column 1 (Identity):** Tagline + social icons. The tagline uses `clamp(1.6rem, 2.8vw, 2.2rem)` with `-0.03em` letter-spacing and weight 500 — intentionally not bold, to differentiate from headings while still commanding attention. Social icons are tighter than the original layout (10px gap instead of 16px) to feel like a unit rather than scattered links.

**Column 2 (Context):** About description with a 28×2px accent bar colored by `var(--persona-accent)`. The accent bar was a refinement suggestion — the column had no visual anchor, making it the weakest of the three. A thin colored bar at the top ties it to the persona system without being heavy-handed.

**Column 3 (Competency):** Horizontal scroll-snap carousel with 6 competency cards.

### The Carousel Implementation

The carousel uses `scroll-snap-type: x mandatory` with cards at `flex: 0 0 82%` width. The 82% is deliberate — it leaves roughly 18% visible of the next card, which is just enough to signal "there's more" without looking like a clipped layout error. A CSS gradient mask (`-webkit-mask-image: linear-gradient(to right, black 0%, black 78%, transparent 98%)`) fades the peek card to zero opacity at the container edge, creating a smooth visual boundary.

Each card has a header row with the competency title and a `01 / 06` counter. The counter uses `font-variant-numeric: tabular-nums` so the numbers maintain consistent width during persona reordering (otherwise "01" and "06" would have different character widths in proportional fonts, causing a visual jump).

### Pagination and Navigation

Pagination dots use Intersection Observer with a 0.5 threshold to track which card is currently visible. The active dot gets `var(--persona-accent)` background and a subtle 1.15x scale transform. Navigation is triple-input: swipe/drag on the carousel itself, click on dots, and arrow key support with proper `tabindex` and ARIA labels.

Post-session addition: left/right chevron arrows flanking the dots, auto-hidden at carousel boundaries (left arrow hidden at card 1, right arrow hidden at card 6). The arrows use the persona accent color at 40% opacity, brightening to full on hover. They reset correctly on persona switch when the carousel scrolls back to position 0.

### Persona Integration

`applyCompetencyReorder()` in `persona-system.js` was updated to support the new carousel. On persona switch: (1) add `.reordering` class for a 200ms opacity dip to 0.3, (2) reorder cards by moving DOM nodes with `carousel.appendChild()`, (3) update all counters to reflect new position, (4) scroll to position 0, (5) reset dots and arrows, (6) remove `.reordering` class after 50ms. The legacy grid fallback was preserved for any future context where the old layout might be referenced.

### Responsive Collapse

Tablet (≤1024px): 2-column grid. Identity and context share a row. Competency column spans full width below with a top border separator.

Mobile (≤768px): Single column with CSS `order` reflow — context (order:1), identity (order:2), competency carousel (order:3). The reorder puts the description first because on mobile, people need "who is this person" before encountering the stylistic tagline.

## The Missing Section: A Morph Choreography Bug

A production bug surfaced mid-session: the entire "By the Numbers" metrics section was invisible. Diagnosis revealed `position: fixed` was being applied by the `.persona-active` CSS rule (which uses `position: fixed` as a hiding mechanism during the morph animation sequence) but never reset by `.morph-complete`.

Root cause: when `.numbers-section` and `.manifesto-section` were added to the morph choreography system earlier in the session, they were included in four of five morph rule groups (`.picker-mode`, `.persona-active`, `.morph-content-in`, `.morph-reverse`) but missed from `.morph-complete` — the final resting state that resets `position` from `fixed` back to `static`.

The section was in the DOM, had `display: block`, `opacity: 1`, and `height: 320px`, but `position: fixed` with `top: 1730px` placed it permanently below the viewport. `scrollIntoView()` couldn't reach it because fixed-position elements don't participate in scroll flow.

**Fix:** Two lines — add `.morph-complete .manifesto-section` and `.morph-complete .numbers-section` to the selector list.

**Lesson:** The morph choreography system has five rule groups that must stay synchronized. Any section added to the morph system needs to be added to ALL FIVE: `.picker-mode`, `.persona-active`, `.morph-content-in`, `.morph-content-in` (the `position: static !important` reset), and `.morph-complete`. Missing one creates an invisible-but-present element that passes every check except visual inspection.

## Metrics Section Redesign: Two-Column INPUTS/OUTCOMES

The biggest structural change: replacing the single 6-card metrics grid with a two-column layout.

### Architecture

The new structure:

```
.numbers-section
  .section-label "(By the Numbers)"
  .metrics-columns (CSS Grid: 1fr 1fr)
    .metrics-col.metrics-inputs
      .competency-eyebrow "INPUTS ↘"
      .numbers-grid (3 cards, persona-specific)
    .metrics-col.metrics-outcomes
      .competency-eyebrow "OUTCOMES ↘"
      .numbers-grid (3 cards, universal placeholders)
      .metrics-tracking-note
```

CSS uses `grid-template-columns: 1fr 1fr` with a `3rem` gap. Each `.numbers-grid` is now `repeat(3, 1fr)` instead of the previous `repeat(auto-fit, minmax(180px, 1fr))` — hard 3 columns since we know the count.

### Per-Persona INPUTS (3 each)

Every persona gets the live commit counter as metric 1 (always dynamic, pulled from GitHub API). Metrics 2 and 3 are chosen per persona for relevance and dynamism:

| Persona | Metric 2 | Metric 3 |
|---|---|---|
| Recruiter | Architectural components (25) | APIs integrated (7) |
| Seeker/Founder | APIs integrated (7) | Live prototypes shipped (3) |
| Practitioner | Architectural components (25) | Accessibility markers (730+) |
| Learner | Prototypes you can try (3) | Components to study (25) |
| Technologist | APIs integrated (7) | Architectural components (25) |
| Inner Circle | Hand-picked assets (70) | Weekends off since February (0) |

Static metrics that never change were cut: "6 Persona lenses" (won't grow), "~6 wks first commit to live" (historical fact), "0 Jira tickets filed" (joke that doesn't evolve).

### OUTCOMES Placeholders

Three universal metrics with dashed values and `opacity: 0.45; border-style: dashed` styling:
1. Persona → Person (visitors who entered via lens, stayed for the human)
2. Connect clicks (the ultimate CTA)
3. Return visits (site as destination)

A subtle italic note: "Tracking begins when Connect goes live." This is honest about the state — better than showing zeroes, which would imply tracking is live but nobody's converting.

### Responsive

Mobile (≤768px): columns stack vertically. Inputs on top, Outcomes below. Mobile small (≤480px): each `.numbers-grid` collapses to single column.

## Session 13: Mechanical Audit Fixes + Mobile UX + Persona Copy Refinement

This was a continuation session after context compaction. Major work happened across three areas.

### Audit Score 96→97: Em-Dash Cleanup Completed

The em-dash violation from the previous audit (present in 10 secondary pages and the blog template) was finally committed and pushed. Result: Authenticity pillar climbed from 94→95, Content from 96→98.

This was a simple fix with outsized impact: replace `&mdash;` HTML entities with literal em-dashes (`—`) everywhere except code snippets and JSON payloads. The audit system detects the HTML entity as "inauthentic" — it implies the copy was generated or templated, not written directly. Real writing uses the character itself.

The fix touched: docs/SITE-WHY.md, docs/ULTIMATE-PERSONA.md, all six persona-detail pages, blog-template.html, and the blog archive listing. Push succeeded without conflicts.

### SVG Fingerprint Bug Fix: Separator Character Matters

The diagram-preview.html SVG violations were baselined at 96 (a high count but acceptable for auto-generated diagram markup). During the audit investigation, I discovered why: the SVG class prefix system uses `:` as a separator (e.g., `diagram:flowchart:element`), but the audit parser was looking for `|` (pipe character) as the separator.

This is an integration bug, not a code bug. Two separate systems were designed with different assumptions about the delimiter format. The audit system expected `|` because that's what the CC Standards documentation specified. The SVG generator used `:` because colons are idiomatic in CSS class naming.

Fix: Update the SVG generator to use `|` instead of `:`. Result: 96 violations → 0 violations on diagram-preview.html. Overall score 96→97.

**Key lesson:** Integration between systems requires exact specification matching. A single character difference can either be invisible (if both systems tolerate it) or catastrophic (if they don't). The audit system documentation now includes a fingerprint separator reference table.

### Mobile CSS Reorder: Identity-Context-Competency Sequence

The mobile layout for the triptych was initially set with `order: 1` on context (description), `order: 2` on identity (tagline), `order: 3` on competency (carousel). This put the elevator pitch description first on small screens.

Kiran caught this during mobile hardware testing: it feels backwards to read the context-heavy description before the identity-establishing tagline. On a 375px viewport, you need the emotional/stylistic anchor before the substance.

Fix: Flipped the order to `order: 1` (tagline), `order: 2` (description), `order: 3` (carousel). The CSS is now:

```css
@media (max-width: 768px) {
  .about-identity { order: 1; }
  .about-context { order: 2; }
  .about-competency { order: 3; }
}
```

This is a 3-line change that affects UX significantly. It's the kind of detail that testing on actual hardware catches immediately, but viewport simulation might miss because the sequence is technically "readable" either way.

### Nav Menu Alignment with Bento Cards

The navigation menu had links to four sections: Causes, Now, Under the Hood, Store. The bento card grid has sections for: About (no menu item), Impact (→Now), Thinking (→Under the Hood), Gallery (no menu item), Learning, Design, AI, Work, Connect (no menu item).

This created a mismatch: two menu items (Causes, Store) had no corresponding bento cards. Two menu items were missing (Under the Hood should be there, and was). Under the Hood already had a coming-soon page, so the menu link was wired but the visual navigation wasn't aligned with the card grid.

Fix: Updated the nav to match the 9 bento card sections: About, Now, Under the Hood, Learning, Design, AI, Work, Gallery, Connect. This is a structural alignment fix — the menu is now a navigation index of the bento grid, not a separate structure.

Code change in `header.html` and corresponding nav click handlers in `bento-cards.js`.

### Dynamic Eyebrow Swap: Persona-Specific Label Override

During the elevator pitch rewrite session, Kiran suggested that "Elevator Pitch" as a label doesn't fit the Inner Circle persona. For close friends, there's no pitch — there's just the unfiltered conversation.

Implementation: Added a `descriptionEyebrow` field to each persona object in `persona-system.js`. If present, this overrides the default "Elevator Pitch" eyebrow label. Usage:

```javascript
{
  id: 'inner-circle',
  name: 'Inner Circle',
  descriptionEyebrow: 'The unfiltered version ↘',
  // ... rest of persona
}
```

The eyebrow update happens during persona morph, in the same DOM operation that updates the description text. For personas without a custom `descriptionEyebrow`, the default "Elevator Pitch ↘" is used.

This is a small addition (4 lines of code, 1 data field per persona) that enables future persona-specific label variations. Other eyebrows (manifesto, metrics, carousel header) could use the same pattern.
