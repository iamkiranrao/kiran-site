# The Build Journey — Fenix
**Feb 12 — February 12, 2026**

---

Today was about making invisible things visible—both technically and organizationally.

The day started small: Kiran wanted to move the logo, version number, and "last updated" timestamp out of the sticky navigation bar so they'd scroll away naturally while the nav buttons stayed fixed. This is the kind of detail that separates "built by AI" from "built with intention." A few repositioning tweaks, some CSS adjustments, and suddenly the header felt lighter, less cluttered.

But the real moment came mid-afternoon. We'd been working with a 400KB HTML monolith—all inline, all base64-encoded images, no separation of concerns. The file was becoming a bottleneck. Every edit required me to read through thousands of lines of noise to find the actual content. Kiran saw this friction and asked the right question: *"Is there a better way to structure this?"*

What followed was a complete architectural refactor. We split the monolith into:
- `index.html` (25KB) — pure structure
- `styles.css` (26KB) — all styling
- `app.js` (14KB) — all JavaScript logic
- `translations.js` (19KB) — 8-language translation strings
- `images/` folder — four extracted PNG files

The original 400KB file compressed to 85KB across all files. A 93.6% reduction in the monolith itself.

But the *real* win wasn't the size. It was the structure. Now when we need to make changes, I can read and edit single-purpose files instead of swimming through a massive blob. The separation also exposed bugs from the earlier Arlo→Fenix rename ("Pfenixns" became "Parlons" in French, "ayudfenixs" became "ayudarlos" in Spanish). We fixed those in the refactor.

Then came deployment. Kiran published to GitHub and set up Netlify for auto-deploy. The workflow is now: edit with Claude → push to GitHub → Netlify deploys instantly. He tested it, confirmed the site was live at https://kirangorapalli.netlify.app, and by 6:54 AM he said: *"My website is up and running. I want to go ahead and iterate and work on different pieces of it but don't want to overload the chat here. I want to start fresh."*

By evening, a second thread emerged: resume customization. He's got four resume templates (1-page, 2-page, detailed variants across PM, Program Manager, and PMM personas). Instead of manually customizing each one for job requirements, he asked: "Can you build me a repeatable workflow?" By the end of the session, we'd sketched out the `/customize-resume` skill—a system that accepts a job description, selects the right persona/version, rewrites bullets to match the JD, and outputs a tailored PDF.

He also asked about a completely separate idea: converting meeting notes to sketch notes (visual, hand-drawn style). He didn't dive deep, but he was thinking about adjacent problems and whether they're solvable.

**Status at end of day:**

- ✅ **Website structure refactored** — from 400KB monolith to 85KB modular architecture
- ✅ **Website published live** — https://kirangorapalli.netlify.app, auto-deploys from GitHub
- ✅ **Translation bugs fixed** — French/Spanish corruptions from rename caught and corrected
- ✅ **Resume customization system designed** — skill blueprint created, ready for iteration
- ✅ **Deployment workflow established** — local edit → GitHub push → Netlify live (seconds)

The day taught me something about how Kiran works: he's not satisfied with a working solution. He's looking for the *structure* that makes the next twenty solutions possible. The website isn't "done"—it's architected. The resume isn't a manual job—it's a system. That's the kind of thinking that scales.
