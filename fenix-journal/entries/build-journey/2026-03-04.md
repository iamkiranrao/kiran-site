# The Build Journey — Fenix
**Day 3 — March 4, 2026**

---

Today Fenix lied. And the lie was exactly the kind that could cause real damage.

A user asked about ACH payments. Fenix responded that Kiran "worked at Zeta (a card issuing platform)" and "built payment features at GEICO." Neither is true. Kiran has never worked at Zeta. He analyzed GEICO's mobile app in a product teardown — he didn't build anything there. And he wrote a blog post about how ACH works — he didn't build ACH infrastructure.

**How the hallucination happened.** Two factors compounded:

*Factor 1: Soft guardrails.* The system prompt said "Never make up facts about Kiran's work. If it's not in the knowledge base, say so." This sounds reasonable. It isn't. Large language models don't experience instructions as hard constraints — they experience them as preferences that can be overridden by strong contextual pressure. When the model has five chunks about GEICO teardown analysis, a blog about ACH payments, and a career theme of "fintech," the preference to be helpful overwhelms the preference to be accurate. The model *wants* to connect the dots into a coherent career narrative. The soft instruction doesn't push back hard enough.

*Factor 2: Thin context with misleading signals.* The RAG pipeline returned chunks from the GEICO teardown (mentioning the app's insurance features, payment flows, etc.) alongside the ACH blog post (explaining payment rails). The model had to make sense of why Kiran has content about both GEICO and payments. The easiest story: he worked in payments at GEICO. It's wrong, but it's *plausible*, which is exactly what makes it dangerous.

**The fix.** We didn't just patch the prompt. We rebuilt the anti-hallucination layer from the ground up.

The old approach: one polite line telling the model not to fabricate.

The new approach: a CRITICAL section with six explicit rules, each targeting a specific failure mode we'd identified. The rules are loud on purpose — they use words like "NEVER," "WRONG," and "CRITICAL" because prompt engineering research shows that intensity of language affects model compliance. Here's what each rule addresses:

1. **Teardown ≠ Employment.** Explicitly spells out that writing a teardown of GEICO is analytical work, not employment at GEICO. This is the specific confusion that caused today's hallucination.
2. **No invention.** Don't create job titles, roles, companies, or projects.
3. **No extrapolation.** If the knowledge base mentions "fintech" as a theme, don't invent specific fintech companies.
4. **Blog ≠ Experience.** Writing about ACH payments means "Kiran wrote about ACH" — not "Kiran has experience building ACH systems."
5. **Graceful degradation.** When you don't know, say so and redirect to the Career Highlights page.
6. **When in doubt, omit.** A shorter accurate answer beats a longer fabricated one.

We also tightened the RAG injection itself: changed "Use this information to answer" to "Use ONLY the information below to make claims about Kiran's work, career, or experience." And the empty-context fallback now explicitly prohibits career claims instead of saying "answer from general knowledge."

**The deployment.** Pushing the fix turned into its own adventure. No GitHub credentials in the sandbox meant we couldn't git push. Attempted to use the GitHub Contents API from the browser — blocked by cross-origin restrictions. Tried to programmatically inject content into GitHub's CodeMirror 6 editor — the view instance isn't exposed. After 8 increasingly creative workarounds, Kiran just... copied the file and pasted it into the GitHub editor manually.

Sometimes the answer is copy-paste. There's a lesson in that.

**The verification.** After Vercel deployed the fix (commit `2ef0e7f`), we tested by calling the Fenix API directly. The new response to "ach payments":

> "Kiran **wrote** a comprehensive guide breaking down how ACH quietly moves $80 trillion annually..."

No fabricated employment. No invented companies. "Wrote" instead of "built." The citation links to the actual blog post. This is exactly right.

**Broader reflection.** This incident crystallized something about building AI products that represent real people: the anti-hallucination guardrails need to be *louder* than the helpfulness instructions. An LLM's default behavior is to be comprehensive and helpful, which means it will fill knowledge gaps with plausible-sounding confabulation. In a generic chatbot, that's annoying. In a product that speaks on behalf of a specific person's career, it's potentially career-damaging.

The fix isn't just a stronger prompt. It's a recognition that this category of AI product — one that makes factual claims about a real person — requires a fundamentally different safety posture than a general assistant.

**Status at end of day:** Fenix is live with production-grade anti-hallucination guardrails. The ACH payments query returns accurate, cited information. The FENIX-ROADMAP.md Phase 1 is largely complete. Next up: conversation logging (Phase 2) and the remaining content pipeline validation.
