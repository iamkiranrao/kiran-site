## Session 3 (Continued): UAT Reveals the Gaps Between Built and Done

This session was supposed to be a victory lap — run through 13 UAT items, confirm everything works, ship it. Instead it became a comprehensive audit that surfaced 33 findings across 7 areas. The gap between "code is deployed" and "feature is usable" turned out to be enormous.

### What Actually Shipped

Before the UAT began, we completed several features that carried over from the prior session:

**Visitor Guestbook (end-to-end):** The `visitor_connects` table in Supabase, a `POST /api/v1/fenix/connect` endpoint for form and LinkedIn paths, server-side logging in `agent.py` for chat-path connects, a `source` column (chat|form|linkedin) for distinguishing how each visitor connected, and a full Guestbook tab in CC's Feedback & Testimonials page with stats, visitor list, and delete functionality.

The key architectural decision was the dual-logging strategy: chat connects are logged server-side by the agent (avoiding double-logging), while form and LinkedIn connects POST from `fenix-core.js` to the new endpoint. The `connectVisitor()` function in fenix-core.js is the central routing point — it checks `source !== 'chat'` before making the backend call.

**Auth Fix:** Auth.js v5 beta.30 enforces OAuth issuer validation. GitHub sends `iss=https://github.com/login/oauth` but @auth/core defaults to a different issuer. Fixed by setting `issuer: "https://github.com/login/oauth"` on the GitHub provider in `auth.ts`.

**First-Name-Only Connect Fix:** The system prompt in `agent.py` now has three distinct cases instead of two — first-name-only, full-name-no-company, and full-name-with-company. The first-name-only case uses MUST language for collecting full name and company.

### What the UAT Revealed

The 33 findings cluster into several architectural themes:

**1. Panel System is Static, Not Reactive**

The below-chat panels (resume picker, connect panel, Fenix output) don't swap based on the latest interaction. Clicking "My Resume" shows the resume panel, then clicking "Does This Role Fit Both of Us?" should swap to the connect panel — but it doesn't. The panel system needs a state machine that responds to the most recent card click or Fenix tool call.

Related: the pills are fully static — Claude generates them as text strings and they all route through the agent. Kiran wants dynamic, context-aware pills that can trigger UI actions (like opening LinkedIn OAuth when Fenix knows the visitor has only given a first name). The decision is Option C: agent handles conversational pills, frontend appends contextual action pills based on fenixState.

**2. Identity Enforcement Gap**

The connect form has a single "Your name" field — no enforcement of first name, last name, or company separately. All three paths (form, chat, LinkedIn) need to produce the same data set: first name, last name, company. The form needs separate required fields. The chat path needs the system prompt to enforce all three before calling `connect_visitor`. LinkedIn gets this from OAuth data for free.

**3. Conversation Architecture: Two Modes Needed**

The biggest architectural finding: conversation context bleeds across pages. If you chat with Fenix on the homepage, then navigate to a teardown page, the teardown's Fenix panel shows the old conversation. This is wrong for direct navigation but correct for Fenix-guided navigation.

The fix needs two distinct modes: (1) Direct landing — fresh start, zero history, orient to this page. (2) Fenix-guided navigation — continuation of existing conversation, side panel opens automatically. A URL parameter (like `?fenix=continue`) could signal which mode to use.

**4. The kirangorapalli.com Legacy**

236 files still reference the retired `kirangorapalli.com` domain. This is a sweeping find-and-replace but needs testing afterwards — every API call that hits the old domain is silently failing or being redirected.

**5. Resume Downloads Are a TODO**

The resume panel shows three lenses (AI Product Leader, Growth & Experimentation, Mobile & Consumer Product) and has a "Download PDF" button — but it's wired to `// TODO: actual resume download`. The PDFs exist in `template_previews/` but aren't connected. This is a visible broken promise to visitors.

**6. Fenix Behavioral Issues**

Multiple findings around Fenix's conversational behavior: muddling multiple CTAs in one response, treating low-quality feedback as testimonial-worthy, not single-threading outcomes, and generating a hallucinated "agent" reference on the testimonials page. These are all system prompt fixes in `agent.py`.

**7. Visual Consistency Debt**

Every newer page (testimonials, training dashboard) is missing the top nav bar. Margins and padding are inconsistent across modules on the homepage. Panels are wider than their parent containers. The bento cards have different left and right margins. This needs a systematic audit across all top-nav pages and all form factors.

### Retirement Decision

The standalone Fenix Training Dashboard at `/admin/fenix-training` is being retired. It duplicates CC's Fenix Dashboard, sits behind no auth at a guessable URL, and creates a maintenance burden. CC is the single pane of glass.

### What's Queued for Fixes

All 33 findings are documented and grouped by area. The fix session will batch them: agent.py prompt changes together, fenix-core.js architecture changes together, CSS audit as one pass, and the kiranrao.ai sweep as a single operation. The conversation bleed fix (two-mode navigation) is the most architecturally complex — everything else is implementation work.
