## The Real Story: Cloudflare Deployment

This is where the session got interesting. Kiran said "can we publish this? It's baked" and what followed was four build failures before we found the root cause.

**The symptom:** Every Cloudflare Workers build failed with "Asset too large." We spent the previous session compressing every PNG in the images directory to under 1MB, bringing individual file sizes well within limits. It kept failing.

**The diagnosis:** The Cloudflare dashboard's Build settings page told the whole story. The build configuration had:

- Build command: None
- Deploy command: `npx wrangler deploy`
- Root directory: `/`
- No `wrangler.toml` anywhere in the repo

Without a `wrangler.toml`, Wrangler was auto-detecting the entire repo root as the assets directory. That meant it was trying to upload 754 git-tracked files totaling ~60MB as static assets — including the entire `command-center/` backend, `prototypes/`, `fenix-journal/`, `archive/`, and everything else that has no business being served as a static site.

The previous successful deploys had been done via `npx wrangler pages deploy .` from Kiran's Mac, which apparently handled asset selection differently (or had a different total size threshold). The GitHub-triggered build with `npx wrangler deploy` was a different codepath with different constraints.

**The fix:** Two files.

1. `wrangler.toml` — Minimal config establishing the worker name, compatibility date, and `[assets] directory = "./"`. This gives Wrangler explicit configuration instead of auto-detection.

2. `.assetsignore` — Gitignore-style file that excludes everything that isn't the static site: `command-center/`, `prototypes/`, `fenix-journal/`, `archive/`, `scripts/`, `.github/`, `*.py`, `*.xlsx`, `*.md`, prototype HTML files, and image originals. This brought the deployable payload from 754 files / ~60MB down to 62 files / ~20MB.

The build completed in 35 seconds. Upload was 62 assets (57 already cached), total worker script was 22KB gzipped. Deployed kiran-site triggers in 1.5 seconds.

**Why this matters beyond today:**

This was a latent infrastructure debt. The site had been deployed manually via Wrangler CLI from Kiran's Mac, which worked but couldn't scale to GitHub-triggered CI/CD. The lack of a `wrangler.toml` meant the deployment was implicitly configured — it worked until the repo grew past whatever threshold Cloudflare Workers enforces on total asset size.

Adding the explicit configuration (.assetsignore + wrangler.toml) is the right fix for two reasons: it makes deploys deterministic (you can see exactly what gets uploaded), and it makes the GitHub → Cloudflare pipeline work the same way as local deploys. No more "it works from my Mac but fails in CI."

**Architecture note:** The `.assetsignore` file syntax follows `.gitignore` conventions. Cloudflare's Wrangler reads it from the assets directory root. It's the static-assets equivalent of a `.dockerignore` — scope what gets shipped, exclude everything else.

## Cloudflare Tunnel: Connecting Cowork to Command Center (and a Security Lesson)

The second half of this session tackled a persistent pain point: the Cowork VM can't reach `localhost:8000` on Kiran's Mac. Every session capture that posts to the CC API fails, and payloads have to be queued as JSON files in `fenix-journal/raw/pending-posts/` and manually flushed from the Mac terminal. It works, but it's friction.

### The Solution: Cloudflare Tunnel

A Cloudflare Tunnel (`cloudflared`) runs on Kiran's Mac, creates an outbound-only encrypted connection to Cloudflare's edge, and routes traffic from a public hostname to the local backend. The CC API becomes accessible at `https://cc.kiranrao.ai` — reachable from Cowork, from anywhere.

**Setup involved:**
1. Installing `cloudflared` (direct binary download — Homebrew wasn't in PATH)
2. Authenticating with Cloudflare (browser OAuth — accidentally selected the `fenixconsulting.ai` zone instead of `kiranrao.ai`)
3. Creating the tunnel (`command-center`, ID: `c85a80c1-...`)
4. Writing config to `~/.cloudflared/config.yml`
5. Adding a CNAME DNS record manually through the Cloudflare dashboard (the CLI route went to the wrong zone)
6. Installing as a persistent launchd service

**Debugging the service was non-trivial.** The `cloudflared service install` command creates a launchd plist that just calls `/usr/local/bin/cloudflared` with no arguments — missing the `tunnel run` subcommand. The service would start, log "Use `cloudflared tunnel run`" repeatedly, and exit. Fix: manually edit the plist to add `<string>tunnel</string><string>run</string>` to the ProgramArguments array. Then the config needed credentials-file paths updated from `/Users/kiran/.cloudflared/` to `/etc/cloudflared/` since the service runs as root.

### The Security Incident

Here's the part that matters most as a build lesson: the tunnel was set up and the DNS record created **before** any authentication was added to the CC API. For approximately 15 minutes, the Command Center backend — containing journal entries, action items, and with a `.env` holding live API keys (GitHub PAT, Anthropic key, Supabase service key, Voyage key) — was accessible on the public internet without authentication.

The risk was low in practice (the subdomain wasn't published, scanners are unlikely to find a new CNAME within 15 minutes), but the process failure was real. The build process gates (Agreement → Gameplan → Pre-flight) existed specifically to catch this class of error. They weren't invoked because the tunnel setup was mentally categorized as "infrastructure plumbing" rather than "a change that alters the security boundary."

### The Fix: API Key Middleware

Added `APIKeyMiddleware` to `main.py` using Starlette's `BaseHTTPMiddleware`:

- Requests from `127.0.0.1` or `::1` (localhost) pass through without auth — the CC frontend keeps working as before
- Health check endpoints (`/health`, `/api/health`) are exempt
- CORS preflight (`OPTIONS`) is exempt
- All other requests require `X-API-Key` header matching `CC_API_KEY` from `.env`
- Missing or wrong key returns `401 {"detail": "Invalid or missing API key"}`

The key is generated via `secrets.token_urlsafe(32)` and stored in `.env`. CLAUDE.md was updated with the key so future Cowork sessions include it in all CC API calls. The session-capture skill's localhost URLs were also updated to the tunnel URL.

### The Process Fix

CLAUDE.md's build process was hardened:
- Gate scope expanded from "code or content" to include "infrastructure changes"
- Gate 3 now includes a **security pre-flight**: "Does this change alter what's exposed to the internet? Does it create, modify, or remove authentication? Does it touch secrets/credentials? If YES: auth first, exposure second. Never the reverse."
- Added a **no-skip list**: infrastructure, auth, DNS, deployment, and secrets changes can never bypass gates, even with "just do it"

### Security Audit Results

A full repo scan found:
- **Cloudflare deployment** is clean — `.assetsignore` properly excludes all sensitive directories
- **Git history** is clean — no `.env` files ever committed
- **File permissions** correct — all `.env` files are `600`
- **One flag:** `prototypes/scannibal/api/.env.local` contains live ElevenLabs and Gemini keys on disk (gitignored, not deployed, but worth rotating)
- **Backend endpoints** are now behind the API key for external access

**Architecture lesson:** Auth goes in first, exposure second. Always. The correct sequence for this session should have been: (1) add API key middleware, (2) test locally, (3) set up tunnel, (4) test authenticated access. We did it backwards. The updated CLAUDE.md now codifies the correct order.

Total effort updated: 70-86 hours combined (up from 64-78 with the two new sections).

### The Password Gate

Kiran wanted to restrict public access during construction. Cloudflare Access (the proper solution) required Zero Trust onboarding with a payment method on file — even for the free plan. Rather than blocking on that, we pivoted to a client-side password gate: `gate.js` injected into all 25 HTML files via `sed`. SessionStorage-based so it persists across page navigation within a tab. Dark overlay matching the site aesthetic.

Not truly secure — someone who views source can find the password. But it's the right tool for "please don't browse yet" at this stage. The password is `workshop2026`, changeable in one line of gate.js. Removal is a single sed command when the site is ready for reveal.

Deployed to production via git push → Cloudflare Pages. The site is now gated.

### The Work Card Strategy Analysis

The biggest strategic output of this session: a deep evaluation of whether the 8 current work cards on index.html are the right 8 for the 6 personas.

**Framework used:** Three lenses — Persona Pull (how many personas does the card serve?), Differentiation (does this exist on every portfolio site?), and Job-to-be-Done Completion (does clicking this card help the visitor complete their actual task?).

**Findings:** Four cards scored weak — Store (zero persona alignment), Causes (no persona JTBD), Learning & Certifications (passive credentials list for one persona), and Studio (blurry differentiation from MadLab). Four potential new cards were identified with research backing — Under the Hood (meta-narrative of building with AI), Frameworks & Thinking Tools (reusable artifacts practitioners share), /Now (Derek Sivers-inspired, what Kiran's working on), and Learn With Me (active mentorship replacing passive credentials).

**Proposed new lineup:** Career Highlights, How I'd've Built It, MadLab (absorbs Studio), Blog & Podcast, Under the Hood, Frameworks & Tools, /Now, Learn With Me. Still 8 cards, but with dramatically better persona coverage and differentiation.

**Decision status: NOT LOCKED.** Kiran was reviewing the analysis when he made the call to capture context and move to a new session with more runway. The card strategy needs his final decisions before it affects the strawman end state work.

### Technical Note

The `gate.js` injection used `sed` to insert `<script src="gate.js"></script>` before `</head>` in every HTML file. First attempt failed because bash history expansion (`!`) interfered with the `if ! grep` logic. Fixed by using `set +H` to disable history expansion and restructuring the conditional with `grep -q || { ... }` instead of `if ! grep`. Minor, but a reminder that shell scripting in Cowork has its own quirks.
