# The Build Journey — Monster Illustration Pipeline + Glass Overlay Redesign
**March 26, 2026**

---

## Two Parallel Workstreams in One Session

Today's session split into two interlocking tracks: (1) generating Pixar/Monsters Inc-style character illustrations in Midjourney for the bento grid cards, and (2) designing the glass overlay system to present text over those illustrated backgrounds. The two informed each other in real time — the overlay design kept changing because the illustrations revealed problems the gradient fallbacks never would.

---

## The Midjourney Two-Channel Discovery

The V6 prompt kit (`docs/BENTO-MONSTER-SCENES-V6.md`) was built around a two-ingredient blend: drop an animal reference AND a monster reference into MJ's Omni Reference box. Kiran discovered through actual use that **Omni Reference only accepts ONE image**. The entire reference strategy had to pivot.

The working approach that emerged: Style References (`--sref`) carry the monster/Pixar aesthetic via `sref-pixar-style.jpg` (an orange fuzzy blob from Monsters Inc). Omni Reference (`--oref`) carries only the animal DNA at very low strength (`--ow 25` instead of the documented 400). This is essentially using the two MJ reference channels as two separate DNA inputs — style ref for "what kind of creature" and omni ref for "shaped like what animal."

The doc rewrite was substantial — 23 prompts updated, all 9 character Drop Zone tables restructured, the conceptual model renamed from "Two-Ingredient Blend" to "Two-Channel Approach." The tuning section was rewritten around `--ow` as the primary dial (tested range 10-100, sweet spot at 25).

**Veteran (bear)** — Orange-brown, blazer with rolled sleeves, rooftop at golden hour. **Tinkerer (meerkat)** — Green-teal, single cyclops eye, welding goggles, sparking contraption. The Tinkerer broke the pattern with a cyclops eye, antennae, and green-teal fur. Future characters planned with: lavender/three eyes (Artist), floppy ears (Connector), etc.

## The Aspect Ratio Mistake

Claude gave HERO (2:1) prompts for every character regardless of default slot. The Analyst was fine — it lives in the HERO slot. But the Tinkerer's default slot is WIDE 3×1, and a 2:1 image cropped to 3:1 showed only a sliver of the character. Kiran caught this: "are you giving me the right dimension as we go along?"

This was a straightforward error — the default slot mapping was right there in the grid layout. Claude should have been generating each character's default slot ratio first, not lazily applying the HERO template to everything. The fix was a corrected mapping table:

## Glass Overlay: Five Iterations to a Solution

The overlay system went through a full design cycle today:

1. **White glassmorphic** (`rgba(255,255,255,0.06)` + blur) — too transparent over illustrated backgrounds, text unreadable.
2. **Dark scrim** (`rgba(20,18,15,0.75)`, blur:0) — readable but still covering character details. Kiran specified the exact values: "glass blur 0 and glass opacity 75%."
3. **Per-card corner positioning** — floating glass boxes at different corners to avoid each character's key details. Got complex, hard to maintain.
4. **Persona-picker bottom strip** — Kiran's suggestion, full-width dark bar at the bottom. Clean but created two new problems: covered MORE of the image than floating boxes (full width vs fit-content), and all overlays at the bottom = visual monotony.
5. **Compact floating glass panels with position variety** — the current approach. `width: fit-content`, `max-width` caps per card, positioned with 4 classes (`pos-bl`, `pos-br`, `pos-tl`, `pos-tr`), backdrop blur at 12px, 85% opacity.

The final approach works because it addresses all the constraints simultaneously: text is readable (dark backing + blur), character details are visible (compact box covers minimal area), and the grid doesn't look repetitive (4 different positions across 9 cards).

**Locked typography values** from this session: Headline `#F5EDE0` (warm white), Eyebrow `#D4A853` (gold), Description `#C8BBA8`. Tags hidden by default, revealed on hover with a `max-height` transition.

## The Preview Page as a Design Tool

`bento-monster-preview.html` was built from scratch as a live testing surface. It started as a simple card render and evolved into a tuning tool with sliders for Object Position Y, Glass Blur, Glass Opacity, and Headline Color toggles. The sliders now affect all overlays globally, making it possible to dial in values across the full grid at once.

The key lesson: visual tuning needs visual tools. The slider-based approach let Kiran drive to exact values ("glass blur 0 and glass opacity 75") by scrubbing rather than trying to guess CSS values in a text editor. This aligns with the locked principle from earlier sessions: "Visual tuning requires visual feedback (browser), not calculations."

## Session 2: Four Characters, One Workflow Revolution, and a Grid Restructure

The evening session produced 4 new monster illustrations (Artist, Connector, Engineer, Explorer) and validated a completely new Midjourney generation workflow that makes the remaining characters and all future aspect ratio variants significantly faster to produce.

### The Incremental MJ Workflow

The biggest technical outcome of this session isn't any single image — it's the workflow discovery. Instead of single mega-prompts, images are built in 3-4 layered runs:

**Run 1 — Stage Lock.** Text foundation (parametric envelope only: aspect ratio, lighting, mood, camera, DOF, negatives) + a bare stage photo in Image Prompts. No style refs, no character. The stage photo should be nearly empty — just light, space, and perspective. The first attempt for the Artist used a detailed studio photo and got cluttered results. The fix: an empty room with golden window light. This "bare stage" principle became the standard for all subsequent characters.

**Run 2 — Rendering Lock.** Add `sref-pixar-style.jpg` to Style References. Add "Pixar 3D animated movie quality, stylized 3D rendering" to text. Critical discovery: the sref + text combo summons monster characters even when you don't want them yet. Fix: add "characters creatures monsters figures" to `--no` list and strip "soft fuzzy fur texture" and "Monsters Inc quality rendering" from text during this run.

**Run 3 — Character Lock.** Add animal omni ref at `--ow 25` + character description in text. Remove creatures from `--no`. Bring back monster-rendering language. This is where the character materializes in the established scene.

**Run 4 (optional) — Refinement.** Props, pose, stance adjustments. Not always needed if Run 3 lands well.

### Characters Produced

**Artist (chinchilla) → Studio → 3:1 WIDE.** Pink-lavender, beret, painting on easel. The first character using the incremental workflow. Validated the entire approach. Stage went from cluttered studio → bare golden room → Pixar room → chinchilla in room → props added. `studiocardwide3_1.png`.

**Connector (quokka) → Testimonials → 1:1 SQUARE.** Warm fuzzy monster in a cardigan, cafe scene. Hit a letterboxing issue (stage photo wasn't square, MJ added black bars at 1:1). Fix: sourced a properly square cafe image. `connector-square-1_1_2.png`.

**Engineer (bulldog) → Under the Hood → 1:1 SQUARE.** Sturdy monster peeking into an engine bay from under a car hood. The "character interacting with a frame" technique worked well — the hood creates a natural compositional border. `engineer2.png` (went through 2 image swaps as Kiran iterated).

**Explorer (gecko/fox) → /Now → 3:1 WIDE (later 4:1).** The first NON-FUZZY character. Smooth gecko-like skin, iridescent teal-green, thin antennae, lean build. This was the monster variation mandate in action. Generated at 3:1 first, but the /Now slot is actually 4:1 (4 columns × 1 row), causing foot cropping. Regenerated at 4:1. `explorer2.png`.

### Grid Restructure

The preview got a significant layout change: Learning expanded from a small cell to full vertical (rows 4-5), Blog and /Now now share row 6 (Blog ~33% width, /Now ~67%). The persona tab system was already built earlier — this session maintained it while reshuffling the grid.

### The Preview Tab System

The complete persona tab system (7 tabs, slot-based architecture, imageMap-driven) was built in this session. The `switchPersona()` function swaps card content, images, overlay positions, and headline fonts per slot. Missing images show gradient fallbacks with `data-need` attributes showing what character and ratio is needed. Image counter tracks progress. This is the visual tracking system for the entire 23+ image generation pipeline.

Bottom-center position class (`pos-bc`) added for Testimonials.
