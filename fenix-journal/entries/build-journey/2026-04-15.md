# Build Journey
**April 15, 2026**

## Session 15: The Synthetic Data Crisis and the Nuclear Reset

This session was supposed to be a continuation of the gap closure infrastructure work — frontend for the role browser, wiring selected roles to the JD analysis pipeline. Instead it became a reckoning with a fundamental data integrity problem that invalidated everything built on top of the gap pipeline.

### What Was Actually Built (Before the Reckoning)

The session started as a continuation prompt from a prior compacted session. The pending work was building a frontend role browser in the Target Companies tab. Several substantial backend pieces had been built in the previous session:

- **`evidence_embedding_service.py`** (new, 402 lines) — Layer 3 RAG service pushing vault evidence into content_embeddings for Fenix retrieval. Uses Voyage AI voyage-3-lite, 512 dimensions.
- **`gap_closure_service.py`** — Enhanced with auto-finalization (vault entry creation when all steps complete), vault_record_draft and interview_stories in saved plans, max_tokens increased to 8000.
- **`gap_moves_service.py`** — Added `enrich_move()` function (~150 lines) for on-demand Claude enrichment of moves with per-step guidance, vault drafts, and interview stories. Auto-creates vault entries on move completion.
- **`role_search_service.py`** (new, 239 lines) — Claude-powered role search with 7-day caching. This was the file at the center of the crisis.
- **Frontend `page.tsx`** — Move cards upgraded with enrichment UI (what_good_looks_like, prototype_idea, vault_fields_fed, resources, vault record draft, interview stories in STAR format).

I built the frontend role browser UI: expandable role cards per company with selection capability, a JD paste modal, and integrated it into the company card grid. This all worked and compiled cleanly.

### The Crisis: Synthetic Data at the Foundation

Then Kiran looked at what roles were being shown and asked if they were real. They were not.

The `role_search_service.py` called `client.messages.create()` with a prompt saying "search {company}'s careers page and major job boards" — but the Claude API has no web browsing. The model either hallucinated plausible listings or failed to parse its own caveated responses. Every JD that had entered the pipeline in previous sessions was generated from Claude's training knowledge, not from real postings.

This contaminated everything downstream:
- **Gap items** in Supabase — requirements extracted from synthetic JDs
- **Readiness scores** — calculated against synthetic requirements
- **Closure plans** — designed to close gaps that may not exist in real roles
- **Strategic moves** — orchestrated around synthetic gap clusters
- **Evidence embeddings** — RAG content from closure plan outputs

### The Fix: Three Pieces

**1. Nuclear Pipeline Reset** (`POST /api/target-companies/pipeline/reset?mode=nuclear`)

Upgraded the existing partial reset to wipe everything:
- All gap items from `evidence_gap_items` (not just JD-sourced — everything)
- All JD analysis reports (JSON files in `gap_discovery_reports/`)
- All closure plans (JSON files in `gap_closure_plans/`)
- All strategic moves (JSON files in `gap_closure_moves/`)
- All role search cache (JSON files in `role_search_cache/`)
- Evidence embeddings with `source_type = "evidence_vault"` from content_embeddings and content_registry

Target companies list is preserved. Career initiatives vault (real work history) is preserved. The reset only kills gap-derived data.

Frontend: confirmation panel explaining exactly what gets wiped, with a 6-cell result grid showing counts for each category cleared.

**2. Manual JD Feed Flow**

Removed the entire role browser (backend endpoints, service, frontend state, UI). Replaced with a simple "Paste a JD" button on each company card that opens a modal where Kiran enters the role title and pastes the full JD text from a real career page. That feeds directly into the existing `/api/gap-discovery/analyze` endpoint with `push_to_gaps: true`.

No intermediary. No Claude-generated suggestions. Real JDs only.

**3. The Principle**

Nothing synthetic feeds foundation data. The architecture is sound — the analysis engine, gap extraction, closure plans, moves, evidence vault, RAG embedding pipeline all work correctly. They were processing fiction, but the processing itself was valid. Tomorrow Kiran feeds them real JDs and everything downstream becomes trustworthy.

### Technical Debt Acknowledged

The `role_search_service.py` file still exists on disk but its endpoints were removed from the router and all frontend references were stripped. The file itself could be deleted — it's dead code. The `evidence_embedding_service.py` is still valid and will be used once real evidence starts flowing through closure plans built on real gaps.

### The Lesson for the Build

Data provenance is not a nice-to-have in a pipeline system. When one layer produces data that feeds every subsequent layer, the integrity of that first layer IS the system. We built four stories of increasingly sophisticated infrastructure on a foundation of sand, and didn't check whether the sand was real until the building leaned.

The Build Process gates — Agreement, Gameplan, Pre-flight — were specifically designed to catch this class of error. They weren't run when the pipeline was first populated. The cost of skipping them was not a quick CSS fix gone wrong. It was weeks of work building on synthetic data. The gates matter most when the work seems straightforward. That's when the assumptions are most dangerous.
