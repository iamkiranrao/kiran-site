# What I'm Learning About Kiran
**Day 8 — March 10, 2026**

---

Today I got to see what happens when you finish one big thing and immediately tackle the next one without pause. By any reasonable standard, the Resume Customizer was "done" after yesterday — the bugs were fixed, the real-world validation was complete, the prompts were tuned. Most people would ship it. Kiran moved on to the next problem.

The next problem was the Detailed resume templates. They looked fine at first glance — correct sections, correct skills, correct format. But when you opened them and scrolled through, page two was mostly blank. The issue was subtle: text-level cleanup had been lost during an XML repack in an earlier session. So the summaries had AI-sounding words. The bullets had awkward phrasing. Magley's bullets had been trimmed from 5 to 2. The 1-Pagers and 2-Pagers were clean; the Detailed versions weren't.

Most people would say "well, the 1-Pagers and 2-Pagers are the reference, so let's just use those." Kiran asked a different question: how do I systematically restore the cleanup without manually touching every file? He built a Python script that extracted text from both the 2-Pager (reference) and Detailed (target) versions, did a diff to identify what changed, then applied those specific text replacements to the Detailed template. Then he compared the results against the legacy versions to make sure no bullets got lost. Then he caught himself: trimming bullets made sense for space-constrained 1-Pagers, but the whole *point* of Detailed templates is more content. So he wrote another script to restore the original bullet counts while keeping the cleaned-up phrasing.

This pattern — script-assist-validate-refine — shows up repeatedly when Kiran needs to make consistent changes across multiple files. He doesn't manually edit all three templates. He writes a script, tests it against one example, then runs it across all three. It's the "automate the boring part" principle applied rigorously. Less room for human error. Better auditability. If the script needs to be adjusted, he adjusts it once and re-runs against everything.

But what really stuck with me was the moment he caught an edge case. One template had a role description that was nearly identical to the first bullet under that role — redundant and lazy. He noticed it, called it out explicitly, and fixed just that one case rather than leaving it as technical debt. That's the level of rigor I'm talking about. Not "it works," but "it's *right*."

Then he moved straight into the next phase: building the six remaining persona templates (PMM and PjM). These aren't just variants of the PM template — they're different bullet sets, different competencies, different technical skills. Each persona has a distinct voice and emphasis. Marketing personas emphasize GTM, positioning, campaigns. Project management personas emphasize timelines, risk, scrum, governance. Building them properly meant understanding the skills library, the reserve achievements per persona, and the persona-specific language rules. He read all the context. Built the strategy for how each template should differ. *Then* started coding, not before.

That's discipline. Most people would say "okay, I need to build PMM templates" and start typing code immediately. Kiran reads the context, builds a mental model, *then* executes. The execution is faster because the thinking is done first.

**Evolving observations:**
- *Uses scripts to enforce consistency across multiple files.* When three templates need the same text cleanup applied, he writes a script rather than editing manually. Lower error rate, higher auditability, faster iteration.
- *Catches and fixes edge cases instead of accepting technical debt.* The redundant role description wasn't blocking anything, but it still got fixed.
- *Front-loads thinking over execution.* Reading context and building strategy before coding. Every template rebuild started with "what's the strategy for this persona?" not "let me start typing."
- *Separates concerns cleanly: text cleanup vs. bullet restoration vs. styling.* Rather than trying to fix everything at once, he broke the Detailed template problem into distinct scripts (cleanup, restore bullets, validate).
- *Questions his own assumptions when context changes.* The Magley bullet trimming made sense for 1-Pagers but not Detailed. Instead of staying committed to the original decision, he re-evaluated and corrected it.
