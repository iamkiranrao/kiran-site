### Session Capture as the Single Post-Session Pipeline
**Date:** 2026-03-21
**Category:** Architecture
**Workstreams:** fenix, command-center, cross-cutting

**Decision:** Enhance session capture to be the single processing pipeline for all post-session updates — journals, action items, Kiran's Journal API posts, and cross-cutting documentation freshness.

**Context:** Cross-cutting documentation (product guides, Tools & Links, Knowledge Library) goes stale whenever modules change, and nothing catches the ripple. Three approaches were considered: startup freshness check (runs when backend boots), end-of-session checklist (runs before wrapping up), and session capture expansion (enhance the existing capture trigger to also update docs).

**Reasoning:** Session capture is already a habit Kiran triggers. Expanding it adds zero new habits. The startup check was shallow — it could detect "something is missing" but not "something is wrong" because there's no baseline to compare against. The end-of-session checklist is unreliable because compaction is invisible — there's no warning signal before context is lost. Session capture, while also dependent on Kiran remembering to trigger it, fits his existing workflow and has access to the full transcript + modified files for ground truth comparison.

**Impact:** Session capture goes from a 1-2 minute transcript + journal operation to a 5-10 minute full processing pipeline. Every capture now produces: Fenix Journal entries, Kiran's Journal API posts, Action Items API posts, ACTION-TRACKER updates, product guide updates, and docs drift flags. The tradeoff is time and tokens per capture vs. silent documentation drift.

**Open Questions:** What happens when multiple sessions pass without a capture? Does drift accumulate to the point where the next capture can't realistically catch up? Should a lightweight startup check still be built as a safety net?

### Objectivity Mandate for Fenix Journal
**Date:** 2026-03-21
**Category:** Principle
**Workstreams:** fenix

**Decision:** Fenix Journal about-kiran entries must include growth areas, friction points, and accurate attribution of problems — not just strengths. Claude's failures must be documented as Claude's, not projected onto Kiran.

**Context:** Kiran reviewed the journal entries and found them one-sided — all strengths, no weaknesses. He asked for more objectivity. Claude's first attempt at criticism projected its own failures onto Kiran (calling Kiran indecisive when Claude was actually the one under-presenting options). Kiran corrected this directly.

**Reasoning:** A journal that only captures strengths is a highlight reel, not a mirror. Growth requires seeing friction. And attribution accuracy matters — if the record says Kiran was indecisive but the actual cause was Claude withholding context, the record is misleading. People will eventually read these entries, so integrity of the record is non-negotiable.

**Impact:** The objectivity requirement is now hard-coded into the session capture skill. Every about-kiran entry must include strengths, growth areas, Claude's own failures, and the interaction pattern between the two. The standard for blame attribution is: if you can identify who caused it, name them.

**Open Questions:** None. This is a principle, not a decision that might change.

### Retiring Content Audit Module
**Date:** 2026-03-21
**Category:** Architecture
**Workstreams:** command-center

**Decision:** Retire the Content Audit module and its product guide. All functionality has been absorbed into Standards & Compliance.

**Context:** The Content Audit module was built as a standalone audit tool. During the Standards & Compliance consolidation, all its checks were rolled into the five-pillar standards framework. The module still exists in the sidebar (with a redirect to Standards) but the product guide was outdated and confusing.

**Reasoning:** Maintaining documentation for a retired module creates noise in freshness checks and confuses the guide index. Better to mark it explicitly retired with a redirect to the current system.

**Impact:** `content-audit.md` guide updated with retired status and redirect to `standards.md`. No code changes — the sidebar redirect was already in place.

**Open Questions:** None.
