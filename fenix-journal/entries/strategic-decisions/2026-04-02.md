# Strategic Decisions
**April 2, 2026**

### Skills-First Over Credentials-First Framing
**Date:** 2026-04-02
**Category:** Product Philosophy | Brand/Identity

**Decision:** Scrapped the certification-card layout (learning.html) entirely and rebuilt as a skills-first interactive explorer (skills.html). Skills are the primary objects; certifications and prototypes are evidence that supports them, not the organizing principle.

**Context:** The existing learning page was 12 certification cards in an accordion layout — credential-first, issuer-first. Kiran's instinct was that this framed him as a credential collector rather than a capability builder. The question "what can this person do?" should come before "what certificates does this person have?"

**Reasoning:** Skills are what hiring managers care about. Certifications are one type of evidence, alongside shipped prototypes, production systems, and project work. By making skills the primary navigation and evidence the secondary detail, the page answers the visitor's real question faster. The data model (skills → sources → links) also makes the page extensible without layout changes.

**Impact:** The entire learning page was rebuilt as skills.html. Navigation across 37 HTML files updated. Sitemap and bento card links updated. The page now serves as both a capability overview and an evidence trail.

**Open Questions:** None — this is locked as the permanent architecture.

### Dual-Mode Visualization: Scannability + Exploration
**Date:** 2026-04-02
**Category:** Architecture | Product Philosophy

**Decision:** The skills page ships with two views — a grouped tag wall (Grid) and a skill-centered force-directed graph (Graph) — with a toggle between them and shared selection state.

**Context:** Six prototypes were built exploring different approaches: constellation map, full force graph, tag wall, dual-mode, hybrid SVG lines, and skill-centered graph. The constellation and full force graph were visually striking but fragile at scale and poor on mobile. The tag wall was extensible and scannable but lacked spatial energy.

**Reasoning:** The tag wall gives the "shape of the skillset in 5 seconds" — grouped by domain, scannable on mobile, easy to update. The skill-centered force graph (where sources are invisible attractor nodes) gives exploration energy — skills that share certifications cluster naturally, revealing cross-domain connections the visitor wouldn't see in a flat list. Having both means each serves its purpose without compromise.

**Impact:** Requires D3.js dependency. The graph is lazy-initialized (only loads when first toggled) to avoid performance cost on page load. Mobile defaults to grid-only.

**Open Questions:** None for architecture. Evidence popup cards are the next feature to build.

### Fenix Widget Shelved Site-Wide
**Date:** 2026-04-02
**Category:** Product Philosophy | Architecture

**Decision:** Removed the Fenix floating action button (fenix-widget.js/css) from all 30 HTML pages and disabled the Fenix subpage banner in persona-system.js. Code preserved for re-enabling.

**Context:** The Fenix banner was overlapping with the skills page's side panel and creating visual clutter (orange border bleeding into content, taking up vertical space that pushed skill content below the fold). Rather than patching around the conflict with CSS hacks, decided to remove it entirely and bring it back when it's ready to coexist cleanly.

**Reasoning:** The Fenix experience isn't mature enough yet to justify the visual real estate it consumes on every page. The banner's chips ("Tell me about Skills", "What should I look at first?") are suggestive but not functional in a way that adds value beyond the persona badge. Cleaner to remove it, focus on making individual pages excellent, and reintroduce Fenix as a coherent assistant experience later.

**Impact:** 30 HTML files modified (CSS + JS references removed). persona-system.js modified (one line commented out). The persona badge ("Viewing as The Inner Circle") still functions — only the Fenix-specific banner and FAB are gone.

**Open Questions:** When to bring Fenix back, and in what form. The current banner format (greeting + chips) may not be the right pattern — a more contextual, page-aware experience might serve better.

### Command Center as Evidence Source of Truth
**Date:** 2026-04-02
**Category:** Architecture

**Decision:** Built a full evidence management system in Command Center that replaces the inline JavaScript data in skills.html. CC becomes the single source of truth for all skills evidence. Data flows out via a static JSON manifest published through the existing git handler — no runtime dependency on CC from the public site.

**Context:** skills.html had 26 evidence sources, 89 skill links, 12 cert details, and 14 item details all hardcoded in JavaScript. Updating evidence required editing HTML files directly. The question was whether to keep data inline, add a simple JSON file, or build full CRUD management.

**Reasoning:** A managed system means evidence updates happen through UI or pipeline automation, not code editing. The static JSON manifest approach gives the best of both worlds — CC manages the data, but the site consumes a committed JSON file with zero runtime dependency. The existing git handler (used by MadLab and teardowns) handles the publish flow. Pipeline hooks on MadLab and teardown deploy mean new evidence is captured at the moment of publication, not weeks later.

**Impact:** 6 new Supabase tables, 25 API endpoints, 4-tab dashboard page, pipeline hooks in MadLab and teardown routers. The skills page will eventually consume from `data/skills-evidence.json` instead of inline data (future migration, not done yet).

**Open Questions:** When to execute the skills.html consumption migration. The page currently still reads inline JS.

### Mind the Gap: Learning as a Managed Backlog
**Date:** 2026-04-02
**Category:** Career Strategy

**Decision:** Built a "Mind the Gap" module inside the Add Skills dashboard that treats skill development as a prioritized, categorized, trackable backlog — not a loose list of "things to learn someday."

**Context:** A PM skills gap analysis revealed critical holes in product analytics, experimentation, and user research. Rather than filing the analysis as a document, Kiran wanted it operationalized: tracked by category, filterable by priority and status, with learning plans and demonstration ideas attached to each item, prioritized by which target personas would be most impressed.

**Reasoning:** Kiran's post-site-launch plan is "I'll be spending time either learning or building." Treating learning goals as backlog items — with status tracking (not-started → researching → in-progress → completed), priority levels, and persona relevance scoring — turns vague career development into a system with the same rigor as product development. The 8-category taxonomy (critical gaps, recognized certs, persona cred, domain specialties, adjacent skills, tool proficiency, frameworks, horizon expanders) ensures coverage across different types of professional development.

**Impact:** New database table, 6 API endpoints, 37 seed items across 8 categories, a new tab in the Add Skills dashboard. Each item carries a learning plan (where to learn, cost, time estimate) and a demonstration plan (how to prove the skill with a portfolio piece).

**Open Questions:** Whether 37 items is too many for a learning backlog — risk of it feeling overwhelming rather than motivating. May need a "focus queue" view that surfaces only the top 3-5 items to work on next.
