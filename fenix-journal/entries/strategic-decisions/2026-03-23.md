### Explorer → Technologist: Replacing a Safety Net with a Real Persona
**Date:** 2026-03-23
**Category:** Product Philosophy

**Decision:** Replace the Explorer persona ("Just Curious / No Agenda") with The Technologist (Ray Turing — "CTO / AI Lead / Tech Lead"). Simultaneously reframe Practitioner from PM-specific to cross-disciplinary ("Product · Design · Data").

**Context:** Kiran noticed that so much of his site showcases AI capability, yet no persona existed for the audience most likely to scrutinize it — CTOs, AI leads, technical evaluators. The Explorer persona was the weakest of the six: it represented the absence of intent, served no measurable outcome, and targeted visitors least likely to use a persona picker in the first place.

**Reasoning:** The decision was driven by a three-part test Kiran developed during the discussion: (1) Does the persona have a question no existing persona covers? (2) Does it need fundamentally different content surfaced? (3) Is it a large enough audience? Explorer failed all three. Technologist passed all three. The UX and data/analytics angles were handled by broadening Practitioner rather than adding a 7th persona, keeping the picker at six options — enough to be meaningful, not so many as to cause decision fatigue.

**Impact:** Affects persona-picker HTML, CSS classes, image assets, meta descriptions, and aria-labels. All updated in this session. The content surfacing strategy for the Technologist persona still needs to be defined — what specific teardowns, prototypes, and case studies get highlighted for this audience.

**Open Questions:** Does the Practitioner copy ("Product · Design · Data") actually resonate with designers in practice, or does it still read as PM-first? The teardown content may need deeper experience-design analysis to serve UX visitors authentically. Also: the Technologist persona promises technical depth — the site needs to deliver on that promise with architecture walkthroughs and MadLab documentation.

### Cloudflare Deployment: From Implicit to Explicit Configuration
**Date:** 2026-03-23
**Category:** Architecture

**Decision:** Added `wrangler.toml` and `.assetsignore` to make Cloudflare Workers deployment deterministic. Only site-relevant files (62 assets, ~20MB) are deployed, excluding command-center, prototypes, fenix-journal, and other non-site directories.

**Context:** Four consecutive build failures with "Asset too large" after switching from manual `wrangler pages deploy .` (from Kiran's Mac) to GitHub-triggered `npx wrangler deploy`. Without a wrangler.toml, Wrangler was uploading all 754 git-tracked files (~60MB) as static assets, exceeding Cloudflare's limits.

**Reasoning:** The root cause wasn't file sizes (we spent an entire previous session compressing images) — it was the scope of what was being deployed. The `.assetsignore` approach was chosen over restructuring the repo (moving site files into a subdirectory) because it's less disruptive and achieves the same isolation.

**Impact:** Unblocks all future GitHub → Cloudflare deploys. Any new non-site directories or file types need to be added to `.assetsignore` to prevent regressions.

**Open Questions:** None — this is resolved.

### Cloudflare Tunnel: Bridging Cowork to Command Center
**Date:** 2026-03-23
**Category:** Architecture

**Decision:** Set up a Cloudflare Tunnel (`cc.kiranrao.ai`) to expose the local CC backend to the internet, enabling Cowork sessions to post journal entries and action items directly without the pending-posts workaround.

**Context:** The Cowork VM runs in a sandboxed Linux environment and cannot reach `localhost:8000` on Kiran's Mac. Every session capture that tries to post to the CC API fails, requiring manual flushing from the Mac terminal. This friction undermined the "one trigger, everything stays current" promise of the session-capture pipeline.

**Reasoning:** Cloudflare Tunnel was the natural choice since the site is already on Cloudflare. The tunnel runs as a persistent launchd service on Kiran's Mac, creates outbound-only connections (no port forwarding), and routes `cc.kiranrao.ai` to `localhost:8000`. Combined with API key authentication, this gives Cowork direct access to the CC API while keeping it locked down from unauthorized access.

**Impact:** Eliminates the pending-posts workaround entirely. CLAUDE.md, session-capture skill, and all CC API references updated to use the tunnel URL with API key header. The backend now has its first authentication layer.

**Open Questions:** Should the CC backend also be protected by Cloudflare Access (zero-trust) for defense in depth? Current API key auth is sufficient for the threat model (personal tool, not multi-user), but it's a single layer.

### Security Pre-flight: Hardening the Build Process Gates
**Date:** 2026-03-23
**Category:** Principle

**Decision:** Expand the three-gate build process to explicitly cover infrastructure changes and add a mandatory security pre-flight check. Infrastructure and security changes can never be skipped, even with "just do it."

**Context:** During the tunnel setup, the build process gates were not invoked because the change was mentally categorized as "plumbing" rather than "a build." This resulted in a ~15-minute window where the CC API was publicly accessible without authentication. The pre-flight check that should have caught this — "does this change alter what's exposed to the internet?" — didn't happen because the gates only applied to "code or content."

**Reasoning:** The failure wasn't in the gates themselves — they were well-designed. The failure was in their scope. Infrastructure changes that alter network exposure, authentication boundaries, or secrets handling are arguably *higher* risk than code changes, yet they were implicitly excluded. The fix: expand the gate scope to include infrastructure, add a security-specific pre-flight item, and create an explicit no-skip list.

**Impact:** CLAUDE.md now includes: (1) expanded gate scope covering infrastructure, (2) security pre-flight as Gate 3 item 3, (3) no-skip list for network/auth/DNS/deployment/secrets changes. This is a permanent process improvement that applies to all future sessions.

**Open Questions:** None. This is the correct fix for this class of process failure.

### Persona Personalization: Look vs. Does (Two-Track Strategy)
**Date:** 2026-03-23
**Category:** Product Philosophy

**Decision:** Persona-driven personalization on index.html will operate in two tracks: Track 1 ("Look") applies visual touches — tagline swaps, card reordering, accent colors, copy adjustments. Track 2 ("Does") introduces a two-column component with persona-exclusive functional unlocks that deliver real value specific to each archetype.

**Context:** The persona picker makes a bold promise: "This site reshapes itself around you." The earlier locked decision scoped all personalization to Fenix. But Kiran pushed for "one layer deeper" — what would make someone feel like the site genuinely adapted? Visual touches alone feel like a theme change. Functional exclusivity (a recruiter gets calendar access, a learner gets mentorship booking) creates the feeling that the persona choice mattered.

**Reasoning:** Kiran evaluates personalization on a "does it change what someone can do?" axis. Track 1 (Look) sets the visual tone — it's necessary but not sufficient. Track 2 (Does) is where the real payoff lives. The two-column component replaces the current generic Fenix intro with a persona-aware left column ("here's what's unlocked for you") and a Fenix intro right column. Each persona gets 2-3 unlocks that are genuinely useful for that archetype — not decorative, functional.

**Impact:** Creates a full playbook: Track 1 is ~14-16 hours, zero dependencies, ready to sprint. Track 2 is ~50-62 hours across three viability tiers. Combined estimated effort: 64-78 hours. Execution order: Track 1 first (all at once), then Track 2 component shell + easy unlocks, then content creation, then calendar integration, then Fenix-dependent features last.

**Open Questions:** Light-theme contrast audit needed for all 6 persona accent colors (only dark theme was audited). The "Bring Your Problem" and "Roast My Product" Fenix modes depend on the Fenix backend being fully wired — these may need "coming soon" stubs initially.

### Index.html Is Universal, Fenix Is Personal (Scope Lock)
**Date:** 2026-03-23
**Category:** Architecture

**Decision:** The main site (index.html) is the same page for everyone. No content reordering, no copy changes, no section swaps based on persona. All persona-driven customization lives in the Fenix chat agent — greeting, suggested prompts, tone, depth, accent color in the widget.

**Context:** The PERSONA-PICKER.md doc previously described full site personalization including "content reordering, contextual framing, and accent colors." Kiran clarified this was aspirational language from an earlier phase. The actual decision is more focused: the site content speaks for itself. Fenix is where personalization has the highest leverage because it's a 1:1 conversation.

**Reasoning:** Building six parallel content experiences for index.html would be high effort for marginal gain — the content sections are the same regardless of who's reading. Fenix, as a conversational agent, can genuinely adapt to who the visitor is in a way static content cannot. The persona picker's value isn't in reshaping the page — it's in reshaping the guide.

**Impact:** This simplifies the site architecture significantly. index.html remains a single, static HTML file with JS-driven persona touches (Track 1), not a templating system with conditional rendering. The two-column unlock component (Track 2) is the one exception — it renders different functional unlocks per persona.

**Open Questions:** Will visitors feel the personalization promise was fulfilled if the page itself doesn't visibly change? Track 1 (tagline, accent color, card order) addresses this — but the real test is whether Track 2 unlocks feel valuable enough to justify the persona selection upfront.

### Site-Building Metrics Over Career Stats for "By the Numbers"
**Date:** 2026-03-23
**Category:** Brand/Identity

**Decision:** The "By the Numbers" section on index.html will showcase metrics about building the site itself (AI tools used, image compression ratios, build hours, etc.) rather than career statistics (years of experience, products shipped, team sizes).

**Context:** Initial proposal was career stats with persona-specific framing. Kiran pushed back: "I think there are other places where you can go deeper into my career work. I think I want to make these metrics specific to the building of this site." He requested objective, research-based rationale.

**Reasoning:** Career stats are commodity signals — every portfolio has "15+ years" and "X products shipped." They're table stakes that confirm credibility but don't create interest. More importantly, career stats already have homes on this site (About section, Career Highlights card, Fenix). Site-building metrics demonstrate capability through the artifact the visitor is standing inside. Transparency into process increases perceived quality (open kitchen effect). The site's thesis is "this is a workshop, not a resume" — career stats contradict that energy. The manifesto script literally ends with "the only metric I care about is whether you reach out."

**Impact:** The "By the Numbers" component (N.2 in PERSONA-PLAYBOOK.md) uses 2 universal anchors + 2 persona-specific stats, all about the site build. Actual numbers need to come from Kiran — some can be mined from git history and docs.

**Open Questions:** What are the actual numbers? Build hours, tool count, compression ratios can be calculated. But some (coffee cups, late-night sessions) need Kiran's input.

**Category:** Product Philosophy

Same count (8), dramatically better persona coverage. Affects the strawman end state, Track 1 card reordering maps, and Track 2 unlock assignments.

**Open Questions:** Kiran hasn't signed off yet. Three of four new cards require content that doesn't exist (architecture write-ups, documented frameworks, /now page). The lineup leans heavily into "workshop" and away from "portfolio" — is the Evaluator persona underserved with only one directly-serving card? Does MadLab actually absorb Studio cleanly, or is there a real distinction worth preserving?
