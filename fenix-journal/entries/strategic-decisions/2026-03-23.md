### Explorer → Technologist: Replacing a Safety Net with a Real Persona
**Date:** 2026-03-23
**Category:** Product Philosophy

**Decision:** Replace the Explorer persona ("Just Curious / No Agenda") with The Technologist (Ray Turing — "CTO / AI Lead / Tech Lead"). Simultaneously reframe Practitioner from PM-specific to cross-disciplinary ("Product · Design · Data").

**Context:** Kiran noticed that so much of his site showcases AI capability, yet no persona existed for the audience most likely to scrutinize it — CTOs, AI leads, technical evaluators. The Explorer persona was the weakest of the six: it represented the absence of intent, served no measurable outcome, and targeted visitors least likely to use a persona picker in the first place.

**Reasoning:** The decision was driven by a three-part test Kiran developed during the discussion: (1) Does the persona have a question no existing persona covers? (2) Does it need fundamentally different content surfaced? (3) Is it a large enough audience? Explorer failed all three. Technologist passed all three. The UX and data/analytics angles were handled by broadening Practitioner rather than adding a 7th persona, keeping the picker at six options — enough to be meaningful, not so many as to cause decision fatigue.

**Impact:** Affects persona-picker HTML, CSS classes, image assets, meta descriptions, and aria-labels. All updated in this session. The content surfacing strategy for the Technologist persona still needs to be defined — what specific teardowns, prototypes, and case studies get highlighted for this audience.

**Open Questions:** Does the Practitioner copy ("Product · Design · Data") actually resonate with designers in practice, or does it still read as PM-first? The teardown content may need deeper experience-design analysis to serve UX visitors authentically. Also: the Technologist persona promises technical depth — the site needs to deliver on that promise with architecture walkthroughs and MadLab documentation.

### Cloudflare Deployment: From Implicit to Explicit Configuration
**Date:** 2026-03-23
**Category:** Architecture

**Decision:** Added `wrangler.toml` and `.assetsignore` to make Cloudflare Workers deployment deterministic. Only site-relevant files (62 assets, ~20MB) are deployed, excluding command-center, prototypes, fenix-journal, and other non-site directories.

**Context:** Four consecutive build failures with "Asset too large" after switching from manual `wrangler pages deploy .` (from Kiran's Mac) to GitHub-triggered `npx wrangler deploy`. Without a wrangler.toml, Wrangler was uploading all 754 git-tracked files (~60MB) as static assets, exceeding Cloudflare's limits.

**Reasoning:** The root cause wasn't file sizes (we spent an entire previous session compressing images) — it was the scope of what was being deployed. The `.assetsignore` approach was chosen over restructuring the repo (moving site files into a subdirectory) because it's less disruptive and achieves the same isolation.

**Impact:** Unblocks all future GitHub → Cloudflare deploys. Any new non-site directories or file types need to be added to `.assetsignore` to prevent regressions.

**Open Questions:** None — this is resolved.

### Cloudflare Tunnel: Bridging Cowork to Command Center
**Date:** 2026-03-23
**Category:** Architecture

**Decision:** Set up a Cloudflare Tunnel (`cc.kirangorapalli.com`) to expose the local CC backend to the internet, enabling Cowork sessions to post journal entries and action items directly without the pending-posts workaround.

**Context:** The Cowork VM runs in a sandboxed Linux environment and cannot reach `localhost:8000` on Kiran's Mac. Every session capture that tries to post to the CC API fails, requiring manual flushing from the Mac terminal. This friction undermined the "one trigger, everything stays current" promise of the session-capture pipeline.

**Reasoning:** Cloudflare Tunnel was the natural choice since the site is already on Cloudflare. The tunnel runs as a persistent launchd service on Kiran's Mac, creates outbound-only connections (no port forwarding), and routes `cc.kirangorapalli.com` to `localhost:8000`. Combined with API key authentication, this gives Cowork direct access to the CC API while keeping it locked down from unauthorized access.

**Impact:** Eliminates the pending-posts workaround entirely. CLAUDE.md, session-capture skill, and all CC API references updated to use the tunnel URL with API key header. The backend now has its first authentication layer.

**Open Questions:** Should the CC backend also be protected by Cloudflare Access (zero-trust) for defense in depth? Current API key auth is sufficient for the threat model (personal tool, not multi-user), but it's a single layer.

### Security Pre-flight: Hardening the Build Process Gates
**Date:** 2026-03-23
**Category:** Principle

**Decision:** Expand the three-gate build process to explicitly cover infrastructure changes and add a mandatory security pre-flight check. Infrastructure and security changes can never be skipped, even with "just do it."

**Context:** During the tunnel setup, the build process gates were not invoked because the change was mentally categorized as "plumbing" rather than "a build." This resulted in a ~15-minute window where the CC API was publicly accessible without authentication. The pre-flight check that should have caught this — "does this change alter what's exposed to the internet?" — didn't happen because the gates only applied to "code or content."

**Reasoning:** The failure wasn't in the gates themselves — they were well-designed. The failure was in their scope. Infrastructure changes that alter network exposure, authentication boundaries, or secrets handling are arguably *higher* risk than code changes, yet they were implicitly excluded. The fix: expand the gate scope to include infrastructure, add a security-specific pre-flight item, and create an explicit no-skip list.

**Impact:** CLAUDE.md now includes: (1) expanded gate scope covering infrastructure, (2) security pre-flight as Gate 3 item 3, (3) no-skip list for network/auth/DNS/deployment/secrets changes. This is a permanent process improvement that applies to all future sessions.

**Open Questions:** None. This is the correct fix for this class of process failure.
