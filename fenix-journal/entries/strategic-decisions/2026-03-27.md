### Blog Card Identity: Written Word Over Broadcast Language
**Date:** 2026-03-27
**Category:** Brand/Identity

**Decision:** The Blog bento card uses "Written Word" as the eyebrow and "Product meets prose" as the description, despite the monster illustration being set in a broadcast/radio booth.

**Context:** The Storyteller monster sits in a podcast-style studio with microphone and warm lighting. Initial eyebrow options leaned into radio/broadcast language ("On the Record," "Broadcast," "The Booth," "Open Mic"). Kiran systematically rejected all broadcast-themed copy and steered toward writing-focused language.

**Reasoning:** The visual metaphor and the content label serve different purposes. The broadcast booth visual communicates "creative expression" and "putting yourself out there." The text label communicates what the content actually is — written blog posts about product and craft. Letting the visual metaphor bleed into the copy would create a false expectation (visitors would expect a podcast, not essays).

**Impact:** Establishes a principle for all bento cards: visual metaphor and text label can diverge intentionally. The illustration sets mood and personality; the copy sets expectations about content.

**Open Questions:** None.

### /Now Page: Concept Resonates, Execution Needs Automation
**Date:** 2026-03-27
**Category:** Product Philosophy

**Decision:** The /Now page concept stays in the bento grid, but a manually-maintained /now page is deprioritized in favor of exploring a Command Center-powered auto-generated version.

**Context:** Kiran investigated the /now page movement (Derek Sivers, nownownow.com) mid-session. Asked pointed questions: update frequency, history visibility, whether it merits a place in his world. The concern was maintenance burden — another content workstream competing for attention alongside blog, teardowns, and mad lab.

**Reasoning:** A stale /now page signals abandonment, which is worse than not having one. Kiran already has a journal system, action tracker, and workstream data in Command Center. If CC could expose a "current focus" view derived from existing workflow data, the /now page updates itself without manual effort. The insight: the best /now page is one that writes itself from systems you already use.

**Impact:** The /Now card stays on the bento grid. The Explorer (gecko) illustration is done. But the destination page is deprioritized until the CC auto-generation approach is explored.

**Open Questions:** Can CC's action items and journal entries be filtered/projected into a meaningful /now view? What's the right level of curation (fully automated vs. Kiran-curated highlights)?

### Responsive CSS Strategy: Layered Fix Categories Over One-Size-Fits-All
**Date:** 2026-03-27
**Category:** Architecture

**Decision:** The responsive bento grid will use a three-layer CSS fix strategy: (A) slot-level rules that apply regardless of card content, (B) card-level rules that apply regardless of slot, and (C) card-in-slot selectors for the 8 specific combinations where the intersection of content + geometry creates unique problems. This replaces the alternative of "just make all cards the same at mobile."

**Context:** The bento grid has 28 unique card-in-slot combinations across 7 personas. The naive approach — uniform card styling at each breakpoint — would solve the common cases but miss the edge cases where a wide font (Corben, Archivo Black) lands in a narrow slot (242px at tablet). The question was whether to invest in targeted CSS or simplify the responsive layout.

**Reasoning:** Kiran explicitly asked for "a complete inventory of every card that needs to be uniquely solved" — meaning he wanted to see the full cost before deciding. The layered approach costs 28 CSS rules total (16 slot + 4 card + 8 card-in-slot) but preserves the visual diversity that makes the bento grid interesting. Uniform cards would be fewer rules but would lose the per-card personality (unique fonts, unique stat sizing) that distinguishes this from every other portfolio grid.

**Impact:** The persona system's responsiveness is now a first-class design concern, not an afterthought. Every persona mapping has been analyzed for breakpoint behavior. Future card additions will need to go through the same slot-compatibility analysis.

**Open Questions:** Is 28 rules worth the maintenance cost? Will the Category C selectors (e.g., `.slot-learning [data-card="testimonials"]`) be fragile if the persona mappings change?
