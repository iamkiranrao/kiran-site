# Strategic Decisions
**April 10, 2026**

---

### Minimum Viable Identity: Enforce First + Last + Company Across All Paths
**Date:** 2026-04-10
**Category:** Product Philosophy

**Decision:** All three connect paths (chat, form, LinkedIn) must produce the same data set — first name, last name, and company — before a visitor can access gated features. No path should accept less.

**Context:** During UAT, the connect form accepted just a first name ("Romeo") and unlocked everything. Kiran's response: "Someone who fills out joe and then gets everything and walks off into the abyss without me being able to know who they are or connect with them in any meaningful way is a waste." The site exists to convert personas into persons — and a first-name-only record has zero relationship potential.

**Reasoning:** This isn't data collection for its own sake. It's the minimum threshold for a meaningful next step. Kiran needs to be able to identify who connected and reach out to them. A first name alone is useless for that. The three-field requirement (first, last, company) gives Kiran enough to find someone on LinkedIn or send a personalized follow-up.

**Impact:** Form needs separate first/last name fields with validation. Agent system prompt enforces all three conversationally before calling connect_visitor. LinkedIn provides all three from OAuth. Every path produces one clean, consistent data set.

**Open Questions:** Should email be optional or required? Currently optional across all paths — LinkedIn may provide it, form offers it, chat doesn't ask proactively.

### Dynamic Pills: Frontend State-Driven (Option C)
**Date:** 2026-04-10
**Category:** Architecture

**Decision:** Pills will use a hybrid approach — the agent suggests conversational pills, and the frontend layer appends contextual action pills based on fenixState.

**Context:** Pills are currently static — Claude generates text strings and they all route through the agent. Kiran observed they don't adapt to conversation context (e.g., if Fenix has only a first name, a pill should offer LinkedIn connect as a one-click shortcut).

**Reasoning:** The agent shouldn't need to think about UI affordances. The frontend already knows what info is missing from fenixState. Keeping these separate means the agent stays focused on conversation while the frontend handles smart shortcuts.

**Impact:** fenix-core.js needs a pill injection layer that reads fenixState and appends relevant action pills after the agent's suggested pills arrive.

### Retire Fenix Training Dashboard
**Date:** 2026-04-10
**Category:** Architecture

**Decision:** The standalone `/admin/fenix-training` page will be retired. Command Center is the single pane of glass for all admin functions.

**Context:** The training dashboard was built before CC existed. It duplicates CC's Fenix Dashboard (conversations, feedback, training gaps), sits at an unauthenticated guessable URL, and creates a maintenance burden.

**Reasoning:** Two dashboards means two UIs to maintain, potential data inconsistencies, and an unnecessary attack surface. CC already does everything the training dashboard does, plus more (quality metrics, failure tracking, guestbook), behind proper GitHub OAuth.

**Impact:** UAT items 10 and 11 (training actions, feedback approval) will be validated in CC instead. The training dashboard HTML and associated API endpoints can be removed.

### Fenix Conversational Discipline: One Message, One Intent
**Date:** 2026-04-10
**Category:** Product Philosophy

**Decision:** Each Fenix response should drive a single outcome. Don't mix feedback collection with Fit Score pitches or project deep-dive suggestions in the same message.

**Context:** When a visitor said "nice site," Fenix captured the feedback AND pitched the Fit Score AND offered project deep-dives — all in one response. Kiran's critique: "If you are trying to drive one outcome, jumbling disparate things together doesn't make sense."

**Reasoning:** Conversational AI has the same UX rules as good product design — one action per screen, one intent per message. Muddling outcomes dilutes all of them. Fenix should stay on the current thread until it resolves (e.g., deepen the feedback into something publishable), then move on.

**Impact:** System prompt in agent.py needs explicit single-threading rules. Also needs a quality gate — generic feedback ("nice site") triggers a deepening question, not a publish offer.

### Two-Mode Conversation Architecture
**Date:** 2026-04-10
**Category:** Architecture

**Decision:** Fenix needs two distinct conversation modes based on how the visitor arrived at a page: direct landing (fresh start) vs. Fenix-guided navigation (conversation continues).

**Context:** During UAT, previous conversation context from the homepage bled into the Amazon teardown page's Fenix panel. The visitor hadn't asked Fenix to take them there — they navigated independently. The old conversation showing up was confusing and wrong.

**Reasoning:** When a visitor navigates independently, they've context-switched. Fenix should too — start fresh, orient to the current page, help them get the most out of it. But when Fenix explicitly guides someone ("let me show you the Amazon teardown"), continuity is the whole point. The signal could be a URL parameter like `?fenix=continue`.

**Impact:** fenix-core.js conversation initialization, localStorage management, and potentially the backend conversation model all need updating. This is the most architecturally complex fix from the UAT.

### Single-Hop Context: Conversation Scoping Across Pages
**Date:** 2026-04-10
**Category:** Architecture

**Decision:** When Fenix navigates a visitor across pages, only carry the previous page's conversation — not the full chain. Homepage → Teardowns → Skills should arrive at Skills with only the Teardowns conversation, not the full history.

**Context:** The two-mode conversation architecture (Session 3) established that `?fenix=continue` signals continuation. But "continuation" initially meant the entire conversation history. Kiran pointed out this was wrong: "If you go from Homepage to teardowns to skills — when you land on skills I don't want the full chat history from the homepage."

**Reasoning:** Each page has its own context and purpose. Carrying the full chain means the Skills page's Fenix has irrelevant context about a Homepage conversation that has nothing to do with skills content. The relevant context is only the immediately preceding conversation — which establishes why the visitor was guided here and what they were exploring.

**Impact:** fenix-core.js now snapshots `lastHopMessages` (last 20 messages from current page) during navigation, and the destination page restores only that snapshot instead of the full message array. The conversation state machine now has three layers: clean slate (direct landing) → previous hop only (guided navigation) → accumulating messages (current page).

**Open Questions:** Should the explored state (cards clicked, panels opened) also be scoped per-hop, or should it remain global? Currently it's global — a visitor who explored resume lenses on the evaluator page will show that in their explored state on any subsequent page.

### Content Collaboration Model: Options Before Implementation
**Date:** 2026-04-10
**Category:** Product Philosophy

**Decision:** When Kiran asks for content updates, Claude should present multiple directions with reasoning for Kiran to choose from — not implement a single direction and present it as done.

**Context:** The "Say Something Real" testimonials heading from Session 3 was written by Claude without discussion. Kiran pushed back: "When I ask for content updates — I want you to riff with me on what they should be instead of assuming you just nailed it. say something real can sound like accusatory."

**Reasoning:** Content carries voice, tone, and emotional register that can't be evaluated from a spec. "Say Something Real" could read as an invitation to authenticity OR as an accusation that the reader isn't being authentic. That ambiguity only surfaces when Kiran reads it — Claude can't reliably judge emotional register for Kiran's audience. The fix is structural: always present options, let the person with the most context (Kiran) make the call.

**Impact:** All content work (copy, headlines, Fenix opening frames, testimonial prompts) should follow the pattern: Claude presents 3-4 directions with distinct tones and reasoning, Kiran chooses, Claude implements the choice. This applies to any user-facing text.

### Pager as Post-Connect Escalation (Pending Agreement)
**Date:** 2026-04-10
**Category:** Architecture

**Decision:** PENDING KIRAN'S AGREEMENT — The `ping_kiran` tool was built to send real-time email notifications when high-intent visitors want to connect live. Built as an agent tool with Resend API integration. However, this was implemented without going through the three-gate build process.

**Context:** The pager concept was discussed in Session 3 as part of the post-connect escalation flow. In Session 4, Claude investigated the infrastructure (Resend API already available), then built the full implementation across agent.py and evaluator-adapter.js without stopping for gates.

**Reasoning for the tool design:** Only triggered when visitor explicitly asks to talk to Kiran, after identity is captured. Sends email with visitor name, company, and reason. Fails silently (notification is best-effort). This aligns with the Session 3 decision that the pager is a post-connect escalation, not an unlock.

**Impact:** Code is written but not deployed. Needs: (1) Kiran's explicit agreement that this should ship, (2) Backend HEAD.lock removal, (3) Re-push to production. The process violation is documented — this is a learning moment about gate discipline, not a feature dispute.

**Open Questions:** Should the pager use Resend email (current implementation) or Pushover (as discussed in the Session 3 design)? The current implementation uses Resend because it was already configured. The Pushover approach would be better for real-time push notifications and could serve as the generic CC notification infrastructure Kiran envisioned.
