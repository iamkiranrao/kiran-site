# Strategic Decisions — March 26, 2026

---

### Single-Scene Composition Over Layered Background+Foreground
**Date:** 2026-03-26
**Category:** Architecture

**Decision:** Each bento card uses a single integrated Midjourney illustration (character inside their environment) rather than separate background and foreground layers composited in CSS.

**Context:** Kiran's initial instinct was a layered composition — blurred scene background + sharp foreground character. Claude argued against it: compositing two separate MJ generations creates style inconsistencies (lighting, color temperature, fur texture) and the seam between "generated together" and "assembled" is a visible AI fingerprint. On tiny 1×1 cards, three layers (scene + character + glass overlay) would compete in ~220px of space.

**Reasoning:** Single-scene generation means one prompt, one consistent lighting pass, no compositing work. Depth of field and character prominence are handled in the MJ prompt itself (shallow DOF, character positioning, environment detail level scaled by card size). This is simpler to produce, more convincing to view, and easier to maintain.

**Impact:** Affects all 23 MJ prompts in the V6 doc. Eliminates an entire CSS compositing layer from the card architecture. Simplifies the per-card styling from 3 z-index layers to 2 (image + overlay).

**Open Questions:** None — this was tested with 3 characters and the approach works.

### Two-Channel MJ Reference Strategy (Style Ref = Monster, Omni Ref = Animal)
**Date:** 2026-03-26
**Category:** Architecture

**Decision:** Monster/Pixar aesthetic comes through Style References (`--sref` at `--sw 200`), animal silhouette comes through Omni Reference (`--oref` at `--ow 25`). One image per channel. Monster ingredient reference files archived.

**Context:** The original plan was to blend two images in Omni Reference (animal + monster, `--ow 400`). Kiran discovered through actual MJ use that Omni Reference only accepts ONE image. The entire reference strategy had to pivot mid-session.

**Reasoning:** The working approach was discovered empirically: putting only the owl image in Omni Reference at very low strength (25 instead of 400) produced the right balance — MJ created a monster that retained owl features (round face, big eyes) while getting the Pixar aesthetic from the style reference. The `--ow` parameter became the main tuning dial (tested 10-100, sweet spot at 25).

**Impact:** V6 doc fully rewritten — 23 prompts updated, all Drop Zone tables restructured, monster ingredient files removed from workflow. The three collected monster reference images (`oref-monster-soft.jpg`, `oref-monster-sturdy.jpg`, `oref-monster-sulley.jpg`) are archived but no longer used in the active workflow.

**Open Questions:** Will `--ow 25` work consistently across different animal references and aspect ratios? Only tested with owl so far. May need per-character tuning.

### Compact Floating Glass Panels Over Full-Width Strips
**Date:** 2026-03-26
**Category:** Product Philosophy

**Decision:** Text overlays on bento cards use compact floating glass panels (`width: fit-content`) positioned at different corners per card, not full-width bottom strips.

**Context:** The overlay system went through five iterations in one session. Full-width bottom strips (persona-picker style) covered too much of the illustration and created visual monotony (all text at the bottom). Floating boxes at different corners avoided both problems but needed per-card position mapping.

**Reasoning:** The bento grid's visual interest depends on the monster illustrations. Covering them with a full-width overlay defeats the purpose. The floating panel approach minimizes coverage (only as wide as the text), and the 4-position system (`pos-bl`, `pos-br`, `pos-tl`, `pos-tr`) breaks the repetitive pattern. Each card's position is chosen based on where the character's key details are.

**Impact:** Affects `bento-monster-preview.html` (current), will propagate to `bento-shapes.html` when finalized. Per-card positioning will need adjustment as each new monster illustration is added — the position map is tied to character composition.

**Open Questions:** The 3×1 WIDE cards (Studio, MadLab) are still tight — 220px row height doesn't leave much room for both character and overlay. May need to revisit row height or overlay minimalism for those specific slots.

### Incremental MJ Workflow Over Single Mega-Prompt
**Date:** 2026-03-26
**Category:** Architecture

**Decision:** Build Midjourney images in 3-4 layered runs (stage → rendering → character → props) instead of cramming everything into one text prompt.

**Context:** The V6 prompt kit treated each generation as a single shot — all scene, character, props, and rendering information in one text string. Kiran discovered through hands-on MJ use that feeding a NYC skyline photo as an image prompt instantly nailed perspective and background better than any text description could. This led to the realization that text should carry parametric constraints (dimensions, lighting, mood, camera) while image inputs carry visual content (scene, style, character form), added incrementally.

**Reasoning:** Isolating variables means diagnosing problems is trivial — if the scene is wrong, fix the stage photo before adding the character. The old approach required dozens of iterations because any change to one element (e.g., props) could cascade into unexpected changes to others (e.g., lighting, character positioning). The new approach produced 4 characters in the time the old approach took for 1.

**Impact:** Replaces the entire V6 prompt philosophy. The 23 existing prompts become text foundations (parametric envelopes) rather than comprehensive scene descriptions. Stage images become a new required input per character. The workflow scales to the remaining 14+ aspect ratio variants needed for persona rotation.

**Open Questions:** Does the approach work at 1:2 portrait (Student) and 2:1 wide (Storyteller) as well as it does at 3:1 and 1:1? The vertical format may need different stage image sourcing strategy.

### Monster Texture Diversification Mandate
**Date:** 2026-03-26
**Category:** Product Philosophy

**Decision:** Remaining monsters (Explorer, Student, Storyteller) must break the fuzzy-blob pattern — explore sleek/smooth skin, translucent/bioluminescent, slimy/amphibian textures. Antennae instead of ears. Varying eye counts.

**Context:** After 5 characters (Analyst, Veteran, Tinkerer, Artist, Connector) all came out as variations of "round fuzzy blob with two eyes," Kiran recognized aesthetic fatigue: "we need to vary up the monsters. antenna eyes, not fuzzy and furry. maybe slimey." The Pixar sref naturally produces fuzzy textures because the reference image is a fuzzy monster — overriding it requires explicit texture language in the text and adding fuzzy/furry/fluffy to the --no list.

**Reasoning:** The bento grid shows all 9 characters simultaneously. If they all share the same texture and silhouette, the grid looks monotonous regardless of color variation. The Explorer gecko (smooth matte skin, antennae) proved the approach works — the Pixar rendering quality survives even when the fuzzy texture is suppressed.

**Impact:** Changes the text foundation for remaining characters. "soft fuzzy fur texture" gets replaced per-character: "smooth matte gecko-like skin" (Explorer), potentially "translucent bioluminescent skin" (Student), "smooth wet amphibian skin" (Storyteller). May need different sref approach or lower --sw for non-fuzzy characters.

**Open Questions:** How far can texture variation go before the characters stop feeling like they belong to the same "family"? The Pixar sref is the unifying thread, but if three characters are smooth and six are fuzzy, does the family cohesion hold?

### Text Foundation Carries Only Parametric Constraints
**Date:** 2026-03-26
**Category:** Architecture

**Decision:** The MJ text prompt sets dimensions, tone, lighting, color temperature, mood, DOF, camera angle, rendering quality, and negative prompts. Scene content comes from image inputs. Character content comes from omni ref + text description added in later runs.

**Context:** Kiran articulated this clearly after explaining the incremental insight: "the text prompt has to set the dimensions, tone, lighting, looks and feel of the image right off the bat. The elements (props, scenes, monsters etc) get built incrementally." Claude initially wrote a foundation prompt that included scene descriptions and character rendering language — Kiran had to correct this twice before the separation was clean.

**Reasoning:** Text is good at parametric constraints (precise numbers, abstract qualities). Images are good at spatial and atmospheric information. Mixing them in text produces unreliable results because MJ interprets text-described scenes loosely. An actual photo of a bare golden room communicates more spatial information than any text description.

**Impact:** The text foundation becomes a reusable template across characters — only the aspect ratio and specific mood/lighting change. Scene images become the primary creative input, which shifts the effort from prompt engineering to image sourcing.

**Open Questions:** None — validated across 4 characters with consistent results.
