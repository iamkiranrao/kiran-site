### Triptych Layout: Compacting the About Section
**Date:** 2026-03-29
**Category:** Product Philosophy

**Decision:** Replace the stacked two-column About section (intro paragraph + divider + 6-tile competencies grid) with a three-column triptych row: identity (tagline + social), context (description + accent bar), competency (horizontal carousel). Reduces vertical scroll significantly while increasing information density.

**Context:** The existing About section consumed substantial vertical space with two distinct sections that didn't visually relate to each other. The divider between intro and competencies created a hard break that interrupted the reading flow. Kiran was inspired by a bento-style three-column layout seen during research, and immediately mapped it onto this problem. The key insight: all three pieces of About information (who I am, what I'm about, what I do) can coexist on a single horizontal row if the competencies become a scrollable carousel.

**Reasoning:** The site's northstar is relational connection, not information hierarchy. Visitors don't need to read all six competencies sequentially — they need to see the first one (the most relevant to their persona) and have the option to explore more. A carousel with persona-ordered cards achieves this with 70% less vertical space than a 6-tile grid. The peek effect (showing 18% of the next card) communicates "there's more" without demanding attention.

**Impact:** Major visual change to the homepage above the fold (on desktop). Affects styles.css (170+ new lines, ~30 removed), index.html (complete About section replacement), persona-system.js (carousel reorder logic). The carousel becomes a new integration surface for the persona system — competency order now visually reinforces persona selection.

**Open Questions:** None — the implementation is complete and in production. The only remaining consideration is whether the mobile reorder (description first, tagline second, carousel third) feels right on real devices vs. the viewport simulation used during development.

### Reconciliation as Maintenance Practice
**Date:** 2026-03-29
**Category:** Architecture

**Decision:** Established a practice of cross-referencing action items against the codebase to identify tracking drift. 23 items were marked completed in a single reconciliation pass. This should become a recurring practice.

**Context:** The Command Center had accumulated 70 action items, roughly a third of which were already implemented but still marked "todo." The drift happened because rapid iteration across sessions (especially the bento grid and persona system) shipped features faster than the tracking system was updated. This is a natural consequence of high-velocity development with session-based AI collaboration — each session focuses on building, not bookkeeping.

**Reasoning:** Stale tracking systems are worse than no tracking system. When a third of your "open" items are actually done, the list loses credibility and prioritization becomes noisy. A reconciliation pass — matching each item against the actual codebase — restores signal. It also surfaces duplicates (5 found this session) that inflate the apparent workload.

**Impact:** Affects how Command Center is used going forward. The CC API's PUT (not PATCH) behavior for updates is now documented knowledge. The reconciliation could potentially be automated — a script that checks file existence, CSS rule presence, or feature flags against the action item descriptions.

**Open Questions:** What's the right cadence for reconciliation? Weekly seems heavy, but monthly allows too much drift. Perhaps trigger-based: reconcile after any session that completes 3+ action items.
