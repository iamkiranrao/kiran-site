### Bento Grid Layout Selected Over Uniform Grid
**Date:** 2026-03-25
**Category:** Architecture

**Decision:** The work card section will use a CSS Grid bento layout (6-column base, variable card spans) instead of the current uniform 4-column grid.

**Context:** Five layout options were evaluated against UX research (cognitive load, paradox of choice, card composition best practices). The bento grid was selected for its ability to communicate hierarchy through card sizing — the flagship card gets more visual weight without the visitor having to sort equal-weight options.

**Reasoning:** NNGroup research shows cards work best when content is heterogeneous and users browse rather than search — both true for this site. The paradox of choice research says the issue isn't item count but whether visitors can orient quickly. Variable card widths solve orientation better than reducing count or reordering alone. The 6-column base is composable: any combination that sums to 6 per row works, making the layout scalable as cards are added or removed.

**Alternatives considered:** Uniform 3-column grid (clean but no hierarchy), featured hero card + 2×2 (hierarchy but rigid), minimal numbered list (compact but no visual richness).

**Impact:** The existing CSS for `.work-grid` (4-column uniform) will be replaced. The persona system will need a `heroCard` property per persona to control which card gets the 4-column hero slot. The bento grid supports 5-8 cards across 2-3 rows within the ~600-800px vertical budget.

### Dimension Flexibility: 60vh Accepted with Valid Rationale
**Date:** 2026-03-25
**Category:** Product Philosophy

**Decision:** The strict 50-55vh target is relaxed to "at or below 60vh" — acceptable to push slightly above if the visual treatment demands it, with a stated rationale.

**Context:** Research recommended 40-60vh for a card section. The initial push was to hit 50-55vh (comfortably within range). After expanding to 9 cards, the grid landed at 60vh. Rather than force further compression at the cost of visual quality, Kiran chose to accept the upper boundary.

**Reasoning:** "I'm happy to be flexible within the limits of reason." The guidelines exist to ensure scannability. If each card has a visual that makes people want to click it and that takes more pixels, the scannability goal is still met through visual appeal rather than compactness alone.

**Impact:** This removes the pressure to hide content (descriptions, visuals) purely for dimension compliance. The final card visual treatment can be designed for appeal first, with dimensions as a guardrail rather than a target.

### Illustrated Character Theme Selected for Card Visual Identity
**Date:** 2026-03-25
**Category:** Brand/Identity

**Decision:** Each bento card will feature a unique Pixar-style monster character illustration as its primary visual identity. Characters are thematically linked (all monsters) but individually distinct.

**Context:** After three rounds of visual treatment (photographic images → flat color headers → gradient backgrounds with emoji icons), none achieved the "in between" visual quality Kiran was looking for. He independently researched bento grid designs overnight, identified five inspirations with specific annotations, and created a mockup in Word showing monster characters on the cards. The Pixar reference connects to existing site elements and Kiran's personal aesthetic.

**Reasoning:** Illustrated characters solve three problems simultaneously: instant card identity (you know which card you're looking at before reading), "same but different" coherence (all monsters, each unique), and boundary-breaking depth (characters spilling outside card edges). This was the strongest recurring pattern across all five inspirations. Monsters specifically were chosen for their human-like features, emotional range, and cultural familiarity (Monsters Inc, Sesame Street).

**Alternatives considered:** Dog breeds (Kiran's dog already appears on the homepage — continuity argument), robots, abstract creatures. Dogs are the fallback if AI generation tools can't produce consistent monster quality.

**Impact:** Requires external illustration generation (Midjourney/Ideogram), new CSS compositing architecture (three-layer: gradient + character + glassmorphism overlay), and character positioning rules for persona-driven card resizing.

**Open Questions:** Which AI tool produces the most consistent style across 9 characters? How many rounds of generation will be needed? Does the Pixar aesthetic align with the professional tone the site needs to maintain?

### Three-Layer Card Compositing Architecture Selected
**Date:** 2026-03-25
**Category:** Architecture

**Decision:** Card rendering will use a three-layer compositing approach: gradient background → character illustration → glassmorphism text overlay.

**Context:** Three implementation approaches were evaluated for handling the persona-driven card resizing (where any card can appear at span 2, 3, or 4). Approach A (one image, CSS crops) lacks control. Approach B (size-specific variants) requires up to 27 assets. Approach C (layered compositing) uses 9 character files with CSS positioning per span.

**Reasoning:** The layered approach makes each layer independently responsive. Gradients adapt automatically. Characters get repositioned via CSS transform/object-position per span class. Text overlay adapts via flex layout. The spill-out effect is cleanly implemented as a separate absolutely-positioned element with overflow:visible and z-index management. Total asset count: 9 character illustrations, not 18-27 variants.

**Impact:** Card HTML structure changes from current (image container + info area) to (gradient layer + character layer + glassmorphism overlay). CSS gains per-card, per-span positioning rules (18 combinations). The persona system continues to work unchanged — it still just reorders DOM and assigns span classes.

**Open Questions:** Does the character layer need to be a single PNG, or could it be an SVG for resolution independence? How do spill-out elements interact with adjacent cards in the grid (z-index stacking)?

### Glass Shelf Architecture for Three-Layer Compositing
**Date:** 2026-03-25
**Category:** Architecture

**Decision:** The glassmorphism text overlay should be a content-sized shelf anchored to the bottom of the card, not a full-card panel.

**Context:** When descriptions were added to all 9 cards, the full-card glass panel left no visible space for the monster character illustrations. The character at z-index 1 would be entirely behind backdrop-filter blur, invisible on smaller cards. This was caught during a spatial feasibility check prompted by Kiran's question about whether the gameplan still worked.

**Reasoning:** Content-sized glass shelf creates two distinct zones: an open staging area (top ~40-50%) for the character illustration and a frosted text shelf (bottom ~50-60%) for content. This produces clearer visual hierarchy — eye catches character first, then reads text. It also makes spill-out effects more dramatic since characters break from open space, not from behind blur.

**Impact:** Required increasing card min-heights (260/280/300px per span), repositioning all character placements, and updating responsive breakpoints. The architecture is now stable for monster illustration integration — only positioning values will need tuning when real PNGs arrive.

**Open Questions:** Are the min-heights right for mobile/tablet? 260px on a full-width mobile span-2 card might feel tall. Needs real-device testing.

### Shape Variety Over Size Variety: The Grid Becomes a Composition
**Date:** 2026-03-25
**Category:** Architecture

**Decision:** The bento grid will use dramatically varied card shapes (wide banners, tall portraits, tiny squares, landscape rectangles) via CSS Grid row-spanning, not just different-sized uniform rectangles.

**Context:** Option D (wider + taller) with three prominence tiers was implemented. The cards were different sizes but fundamentally the same shape — all rectangles with the same proportions at different scales. Kiran identified the problem: "we need more visual breaks... we are at risk on these cards being not so different from my original design. We need cards of different shapes, wide, long square etc."

**Reasoning:** A bento grid's defining characteristic is shape heterogeneity, not just size heterogeneity. Cards spanning 2 rows create tall/portrait shapes. Single-row-wide cards create banners. The combination of column span and row span produces five genuinely distinct aspect ratios (2.1:1, 1.6:1, 1:1, 3.2:1, 6.5:1). This variety is what makes the layout feel composed rather than arranged. It also solves the character illustration problem: different shapes demand different camera framings, creating natural visual variety without forcing it.

**Alternatives considered:** Staying with Option D's uniform rectangles at different sizes. Rejected because it still read as "a grid with big and small items" rather than a composed layout.

**Impact:** Requires explicit CSS Grid placement (no auto-flow). Changes every card's aspect ratio, which invalidates the V5 scene prompt dimensions. Character illustrations must now be designed for each card's specific shape — the card is a window, and the window determines the scene.

**Open Questions:** Does the persona system's card reordering still work with explicit grid placement? The current prototype has fixed positions — persona-driven reordering would need to swap which card occupies which grid slot, not just reorder the DOM.

### Card Shape Determines Character Design, Not Vice Versa
**Date:** 2026-03-25
**Category:** Product Philosophy

**Decision:** Monster character illustrations are designed from the card shape outward — the card's aspect ratio determines the character's pose, camera framing, and activity. Not the other way around.

**Context:** Kiran articulated this through specific examples: How I Built It (4.4:1 ultra-wide) should show a monster that has broken down a machine with pieces scattered across the width. Learning (1:1 tiny square) should show a monster lying on its belly reading. Testimonials should use a waist-up close crop for intimacy. The card's proportions are the creative constraint that generates the scene concept.

**Reasoning:** This is the shift from "mascots on cards" to "scenes in windows." When you design a character first and then try to fit it into a card, you get scaling problems (too small in wide cards, cropped in narrow ones) and dead space. When you design from the card shape first, the character's pose and framing naturally fill the space because they were conceived for it. Wide cards get panoramic scenes with horizontal activity. Tall cards get standing/portrait compositions. Tiny cards get intimate close-ups.

**Impact:** The V5 scene prompt document was updated with scene concepts per card, but the exact dimensions need refreshing for the bento-shapes.html layout. Each Midjourney prompt uses `--ar` matched to the card's stage area ratio. The compositing approach shifts from PNG character cutouts to full scene renders that become the card's background layer.

**Open Questions:** None — this is a locked design principle. The execution details (exact prompts, exact aspect ratios) will evolve, but the principle holds: shape determines scene.

### Explicit Grid Placement Over Auto-Flow
**Date:** 2026-03-25
**Category:** Architecture

**Decision:** Every card in the bento grid uses explicit `grid-column: start / end` and `grid-row: start / end` placement. No CSS Grid auto-flow.

**Context:** First attempt at the shape-varied grid used `grid-column: span N` and `grid-row: span N`, relying on CSS Grid's auto-placement algorithm. The algorithm left gaps wherever column counts didn't sum to 6 within a row — three empty cells appeared in the layout.

**Reasoning:** Auto-placement works for uniform grids where all items are the same size. For bento layouts with heterogeneous shapes and row spans, the algorithm can't solve the 2D packing problem reliably. Explicit placement guarantees gap-free layouts at the cost of manual position management.

**Impact:** Adding or reordering cards requires updating the explicit grid coordinates, not just appending to the DOM. The persona system will need a mapping of card ID → grid position per persona, rather than just DOM reordering.

**Open Questions:** How complex does the persona mapping become? With 6 personas × 9 cards × explicit positions, that's potentially 54 position rules. Is there a simpler abstraction?
