Multimodal AI — A model that processes multiple types of input (text, images, audio) in a single request. Scannibal sends a photo and a text prompt together — the model sees both simultaneously.
Gemini 2.5 Flash — Google's fast multimodal model optimised for high-throughput, low-latency tasks. Flash processes image + text queries in under 2 seconds — fast enough for a real-time scanning experience.
Prompt Chain — A sequence of instructions that shapes how the model interprets an image. Each Scannibal mode has its own prompt chain tuned for that domain — the same photo of a label produces very different analyses depending on which mode you're in.
Serverless Function — A piece of code that runs on demand without a persistent server. Vercel spins up a function when a scan request arrives, processes it, and shuts down. You pay per invocation, not per hour.
Structured Output — Constraining the model to return data in a specific JSON format instead of free-form text. Scannibal's prompts enforce fields like threat level, flags, and recommendations so the app can render them consistently.

Scannibal — Vision AI Engine

The multi-modal pipeline behind the scanner. Gemini 2.5 Flash processes photos through 8 domain-specific prompt chains — each one tuned for a different type of visual input, cross-referenced against the user's personalised safety profile, and returned as structured JSON in under 2 seconds.

The Challenge

Most vision AI demos are one-trick — they classify an image or extract text from it. Scannibal needs to do something harder: take the same type of input (a photo) and produce fundamentally different analyses depending on the context. A photo of a food label means something different when you're checking for pet toxicity than when you're checking for medication interactions. The visual recognition is the easy part. The hard part is building a prompt architecture that routes the same image through the right domain knowledge.

The second challenge is personalisation. A generic label scan is interesting once. A scan that knows you're on blood thinners and flags the vitamin K in that supplement — that's useful every time. Every prompt chain injects the user's safety profile as context, so the same label photo produces different flags for different people.

The Pipeline

Every scan follows the same flow: the app captures a photo, selects the active mode, loads the user's relevant profile (allergies for menu scans, pet types for toxicity scans, medications for supplement scans), and sends all of it as a single multimodal request to a Vercel serverless function. The function assembles the domain-specific prompt, attaches the image as base64, and forwards it to Gemini 2.5 Flash.

Gemini processes the image and text together — it can read ingredient lists, identify produce by appearance, parse nutrition tables, and recognise plant species all within the same model. The prompt constrains the output to structured JSON with consistent fields: a threat level (clear, caution, danger), an array of specific flags with explanations, and actionable recommendations.

Why Gemini over GPT-4V or Claude? Three reasons. First, Gemini Flash is fast — under 2 seconds for a photo + 2000-token prompt, which matters for a scanning app where you're pointing your camera and waiting. Second, the free tier is generous enough for a beta (roughly 500 scans per day). Third, Gemini's image understanding is strong on text-in-images — ingredient lists, nutrition labels, menu text — which is 5 of the 8 scan modes.

Prompt Engineering

Each of the 8 modes has its own prompt chain with domain-specific knowledge baked in. The Menu Analysis prompt includes a comprehensive knowledge base for 5 religious dietary traditions — not just "no pork for halal" but the nuance: gelatin sourcing, alcohol in sauces, rennet in cheese, lard in pastry, cross-contamination in shared fryers. The Supplement Checker prompt knows 10+ medication interaction classes and can flag that fish oil and blood thinners don't mix.

The prompts don't just ask Gemini to "analyse this image." They give it a role (the butler concierge persona), a constraint set (only flag what's relevant to this user's profile), a knowledge base (domain-specific facts the model can reference), and an output schema (structured JSON with specific fields). The result is that the same underlying model produces very different — and very specific — analyses depending on the mode.

Example: Prompt Structure (simplified)

Role → "You are a sophisticated butler concierge..."
Profile → User's allergies, medications, pet types, skin conditions
Domain knowledge → Mode-specific facts (e.g., 60+ hidden sugar aliases for Grocery Decoder)
Image → Base64-encoded photo from camera
Output schema → { threatLevel, flags[], recommendations[], butlerNarrative }

Infrastructure

The API runs as a single Vercel serverless function (POST /api/scan). It accepts a mode identifier, an image, and the user's profile — then routes to the correct prompt builder. Rate limiting is enforced at 50 scans per day per device during beta, tracked server-side. The function cold-starts in under 500ms and processes a scan end-to-end in 1.5–3 seconds depending on image complexity.

There's no database. The user's profiles and scan history are stored locally on the device using React Native's AsyncStorage. The API is stateless — every request contains everything needed to produce a result. This keeps the architecture simple, the costs low, and the privacy strong: no user data is ever stored on a server.

Why stateless? A food safety scanner needs to work reliably and cheaply at scale. No database means no connection pooling, no migration headaches, no data breach surface area. The trade-off is that scan history doesn't sync across devices — acceptable for a v1 where most people use one phone.
